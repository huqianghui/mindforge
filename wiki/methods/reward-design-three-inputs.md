# Reward 设计三份输入与两本账分家

## 摘要

Reward 设计不是写规格，是做实验。这个工作流回答"训练用的分数该怎么定"：第一天就要凑齐**三份输入**——业务属性（问客户）、数据分布（扫标签）、基线能力（一次校准跑）；只用第一份（按业务直觉拍权重）是 video2frames 原始 reward 错配的根源。核心原则是**两本账分家**：验收分逐字节不动（历史可比、业务汇报口径），优化 reward 单独按信噪比设计（拆分 judge 降分母、饱和分量约束化还分子、确定性规则分白送零噪声信号）。业务对齐靠**盲选 A/B 反推权重**（judge 与业务盲选一致率 ≥85% 才获得代理资格），落地形态是一次一小时的共商工作坊 + 训练前锁版。改完必须重走 [[pre-run-three-number-accounting]] 三个数——判断式翻转为"值得跑"才开训。

## 适用条件

- **前置依赖**：有带标签/GT 的评估数据集；能跑一次小规模（约 100 条）baseline 校准；业务方可投入约一小时
- **适用场景**：任何以 LLM judge / 混合评分为 reward 的文本优化（APO、SkillOpt、RL 共用——reward 是三者共享的地基）；三个数算账给出"停"且归因指向 reward 结构时
- **不适用场景**：纯可验证 reward（如数学答案精确匹配）无 judge 噪声问题，不需要拆分与盲选校准；一次性实验（不值得付共商工作坊成本）

## 步骤

### Step 1: 凑齐三份输入

- **输入**：客户访谈、数据集标签、一次约 100 条的 baseline 校准跑
- **操作**：① 业务属性——字段重要性、漏报/误报不对称代价、下游消费方式（问客户）；② 数据分布——类别均衡、正例稀疏度、难度散布（扫标签，零训练成本）；③ 基线能力——各分量饱和度、judge 噪声 σ_noise（校准跑，几美元）
- **输出**：三份输入清单；每个候选分量的"饱和度 + 稀疏度 + 噪声"画像
- **判断标准**：三份都有实据；大半"问客户的问题"应转为测量动作——答案在数据里

### Step 2: 两本账分家

- **输入**：现有验收口径
- **操作**：验收分原公式逐字节不动（历史 run 可比、最终 faceoff 仍用它裁决）；另设优化 reward 只喂训练循环（候选排序 / 门控 / 反思桶），完全按信噪比设计
- **输出**：两个明确分工的评分函数
- **判断标准**:验收口径零改动；业务汇报永远用验收分和 hard 比率，优化量纲不出场——沟通层同样分家

### Step 3: 按信噪比改造优化 reward 结构

- **输入**：Step 1 的分量画像
- **操作**：四个决策逐一对症——① 合并 judge 拆成分字段半独立分（噪声按约 1/√3 收缩，分母 σ_d 降）；② 饱和分量从加权项改乘法约束（"不许退步"，死权重还给有信号分量，分子 δ 恢复）；③ 极稀疏正例诚实处理（正例撑不起统计就做独立 canary 回归套件，不进主 reward）；④ 已有硬性风格规则用正则/词数确定性打分纳入（零噪声梯度素材，白送）
- **输出**：reward v2 设计稿（结构定、权重为初始值）
- **判断标准**：每个结构决策能指到一个实测病灶；judge 噪声占 σ_d 大头时加"每字段 3 次取中位数"杠杆

### Step 4: 共商工作坊 + 盲选校准（约一小时，业务到场）

- **输入**：reward v2 设计稿、10~20 对真实输出
- **操作**：开场 10 分钟讲两本账不讲公式（拆掉"分数要变"的误解）；盲选 A/B 20 分钟——只问"哪个更好"不解释来源，逐对记录；用分数阶梯样例（0.9/0.7/0.4 各长什么样）锚定量纲；确认不对称惩罚比例；落字为凭（reward 设计文档加"业务确认记录"）
- **输出**：业务偏好记录 + 确认签字
- **判断标准**：judge 排序与业务盲选一致率 ≥85% 才获得"代理业务"资格（<85% 先修 judge rubric 再来）；不问权重、问偏好，从偏好反推权重

### Step 5: 校准跑 + 重走三个数 + 锁版

- **输入**：v2 reward、新 baseline
- **操作**：新量纲下重定 hard 阈值（按业务盲选的"可交付线"对应分位数）；重录 baseline、重测 σ/σ_d/δ_remain；判断式翻转为"值得跑"才锁版开训
- **输出**：锁版的 reward v2 + 开跑决策
- **判断标准**：一次谈透、训练前锁版——大规模训练后再改分数等于重付一遍训练成本；此后任何改版重触发同一流程

## 决策点

| 条件 | 选择 | 理由 |
|------|------|------|
| 分量饱和（如 0.94+）无梯度信号 | 加权项 → 乘法上限约束 | 语义从"继续加分"变"不许退步"，死权重还给有信号分量 |
| 正例极稀疏（如 0.2%，各切分为 0） | 独立 canary 回归套件，不进主 reward | 个位数样本进 val 反而扭曲 gate；单独汇报召回更诚实 |
| 正例稀疏但业务关键且量够 | 过采样进 train/val + 不对称惩罚 | 漏报比误报罚得重，比例找客户确认 |
| σ_noise 占 σ_d 大头 | judge 每字段 3 次取中位数 | 3 倍 judge 调用换 √3 收缩，通常比扩 val 划算 |
| prompt 有硬性规则但 reward 不测量 | 加确定性 rule_compliance 分量 | 正则/词数可算、零噪声，免费的可优化空间 |
| judge 与盲选一致率 <85% | 先修 judge rubric 再校准 | judge 无代理资格时，训练循环的百万次打分方向不可信 |

## Claims

### Claim: reward 设计不是写规格是做实验——第一天就需要三份输入，只用业务直觉拍权重是可在设计当时发现的缺陷

- **来源**：[[SkillOpt系列04：APO×SkillOpt联合展望——先探索后精修的两段式管道与选型算账方法]]
- **首次出现**：2026-07-15
- **最近更新**：2026-07-15
- **置信度**：0.85
- **状态**：active

> video2frames 原始 reward（0.2·scene + 0.2·courier + 0.6·judge）只用了业务属性一份输入。courier 正例 0.2%（5,847 条里 13 个）**扫一遍标签就能发现**（零训练成本），发现即可推出"0.2 权重 + 对称匹配 = 常数分量"；scene 饱和在 0.94 要跑了才知道，但一次 100 条校准跑只要几美元——与 149 万 token 无效训练相比可忽略。后果是信噪比双向恶化：0.4 死权重把可观测 δ 结构性打六折（信号压缩），σ_d 里剩下的几乎全是合并 judge 噪声（噪声占比升高）——"停"字一半是任务天花板，另一半是 reward 结构自己造成的。

### Claim: 两本账分家——验收分逐字节不动，优化 reward 单独按信噪比设计

- **来源**：[[SkillOpt系列04：APO×SkillOpt联合展望——先探索后精修的两段式管道与选型算账方法]]
- **首次出现**：2026-07-15
- **最近更新**：2026-07-15
- **置信度**：0.8
- **状态**：active

> 用一个"按业务直觉加权"的函数同时充当验收分和训练信号是根本缺陷。分家后：验收分不动（历史可比、汇报口径、最终 faceoff 裁决）；优化 reward 只喂训练循环，四个结构决策逐一对症实测病灶——拆分 judge（0.45/0.20/0.10，噪声 1/√3 收缩）、饱和分量约束化（scene 错 ×0.5、courier 误报 ×0.3）、canary 套件、rule_compliance 0.25（正则/词数确定性打分，prompt 要求它们但 reward 从未测量它们）。预期分子放大 + 分母减半（σ_d 0.12→0.06），让 APO 排序和 SkillOpt 门控同时重新可分辨。沟通层同样分家：汇报永远用验收分和 hard 比率（"可直接交付比例从 59% 到 X%"），soft_opt 是工程内部量纲不出场。

### Claim: 盲选 A/B 反推权重——不问权重问偏好，judge 一致率 ≥85% 才获得代理业务资格

- **来源**：[[SkillOpt系列04：APO×SkillOpt联合展望——先探索后精修的两段式管道与选型算账方法]]
- **首次出现**：2026-07-15
- **最近更新**：2026-07-15
- **置信度**：0.8
- **状态**：active

> 业务说不出"detail 该占 0.45"，但对着两条真实输出选 A/B 毫不含糊。judge 排序与业务盲选一致率 ≥85%，judge 才获得"代理业务"的资格——此后训练循环里的百万次打分都不用业务再看。落地是一次一小时共商工作坊（讲两本账 10 分钟 + 盲选 20 分钟 + 阶梯样例锚定量纲 + 落字为凭），训练前锁版；业务一小时的投入，换掉的是训练完才发现优化错方向的整轮返工——"分数对话是全项目最便宜的保险"的操作化版本。

## 实践记录

### Practice: 2026-07-15 — video2frames reward v2 设计稿

- **应用场景**：三个数算账判"停"后，溯源到 reward 结构错配，产出 v2 完整规格（judge 三字段拆分 + 乘法约束 + rule_compliance 六项检查清单 + courier canary 套件 + hard 分位数重校准）
- **实际结果**：部分成功——设计稿完成且每个结构决策对应实测病灶，但权重数字（0.45/0.20/0.10/0.25）和惩罚比例（×0.3/×0.2）是工程推理的初始值，盲选校准与新 baseline 校准跑尚未执行
- **偏差与调整**：courier 未走"过采样"分支——本数据集各切分正例数为 0，13 条正例改做独立 canary 套件
- **经验教训**：移植时逐字节兼容评分（为可比性，本身正确）会忠实继承上游 reward 缺陷，直到 held-out 实测才照出来——移植 ≠ 免检
- **置信度影响**：设计稿级证据，Claims 置信度 0.8~0.85；待盲选校准 + 判断式翻转实测后可上调
