# 先算账再开跑——三个数判断式（σ_d / δ_min / δ_remain）

## 摘要

在启动一轮 prompt/skill 优化（尤其是 APO→SkillOpt 两段式管道的第二段）之前，用三个数回答"这轮训练值不值得跑"：**σ_d**（配对差值的散布，决定评估噪声下限）、**δ_min = 2×σ_d/√n**（门控/比较在给定 val 大小下能分辨的最小真实差异）、**δ_remain**（当前候选距任务天花板的剩余提升空间，实测估计非理论推导）。判断式：`δ_remain > δ_min → 跑；δ_remain ≤ δ_min → 停（瓶颈不在 prompt）`。三个数全部从已有 eval 产物 `results.jsonl` 免费算出，时点在上一段优化结束后、下一段开跑前——几分钟的算术可以省掉一次百万 token 的无效训练。video2frames 实测把账走完的答案是"停"，且矛头指回 reward 结构（见 [[reward-design-three-inputs]]）。

## 适用条件

- **前置依赖**：已有至少一次逐任务打分的 eval 产物（每任务一行分数）；评分函数与 target 模型在算账与后续训练之间保持不变（换任何一个都要重估）
- **适用场景**：接续型优化决策（APO best 之后要不要接 SkillOpt 精修）、任何"gate 靠 val 分数 accept/reject"的文本优化开跑前、扩 val 还是加采样的预算分配
- **不适用场景**：没有逐任务分数（只有单个汇总分）时算不了 σ_d；评估集与训练目标分布严重不一致时，算出的数没有外推意义

## 步骤

### Step 1: 两次 eval 测 σ 与 σ_d

- **输入**：同一 val split 上 baseline 与种子候选（如 APO best）各一次逐任务打分（`results.jsonl`）
- **操作**：按任务 id 对齐两列分数；对单列求 std 得 σ（任务难度散布）；逐任务相减得差值列，对差值列求 std 得 σ_d。可选第四个数：同一个 skill 评两遍、逐任务差值求 std → σ_noise（纯 judge 噪声下限）
- **输出**：σ（video2frames ≈ 0.20）、σ_d（≈ 0.12）、可选 σ_noise
- **判断标准**：σ_d 显著小于 σ（配对相减剔除了任务难度这个共同因素）；若两数接近，说明两候选行为几乎不相关，需检查对齐是否正确

### Step 2: 算门控分辨率 δ_min

- **输入**：σ_d、计划的 val 大小 n
- **操作**：`SE = σ_d/√n`，`δ_min ≈ 2×SE`（95% 置信）；反解 `n ≈ (1.96×σ_d/δ)²` 可得"想分辨 δ 需要多大 val"
- **输出**：δ_min（σ_d=0.12、n=100 → δ_min≈0.024）
- **判断标准**：小于 δ_min 的真实提升在门控眼里与噪声无法区分——accept/reject 退化成抛硬币

### Step 3: 实测估计剩余空间 δ_remain

- **输入**：种子候选相对 baseline 的实测增益、各 reward 分量的饱和度
- **操作**：从分量分解看还有多少可动空间（饱和分量≈0 贡献），估计天花板；δ_remain = 天花板 − 已实现增益
- **输出**：δ_remain 区间估计（video2frames：APO best 已 +0.018，分类分量 0.94~1.00 饱和，天花板 ≈ +0.02 → δ_remain ≈ 0.002~0.02）
- **判断标准**：是实测估计不是理论推导——必须以真实 eval 分量数据为依据

### Step 4: 代入判断式并决策

- **输入**：δ_remain、δ_min
- **操作**：`δ_remain > δ_min → 跑`；`δ_remain ≤ δ_min → 停`。停时反算翻案成本作证据（分辨 +0.01 需 n≈(1.96×0.12/0.01)²≈550 条 val）
- **输出**：跑/停决策 + 成本账
- **判断标准**：决策有数字凭据；"停"时矛头应转向 reward 结构或任务天花板，而非换个优化器再试一次

## 决策点

| 条件 | 选择 | 理由 |
|------|------|------|
| δ_remain > δ_min（付得起的 n 下） | 接下一段优化 | 门控/比较能分辨剩余空间，训练有统计学意义 |
| δ_remain ≤ δ_min | 停，转向 reward 结构或数据 | gate 抛硬币，烧 token 原地踏步 |
| σ 与肉眼分数分布对不上 | 先画分数直方图 | σ 可能被"主峰+低分尾"混合结构撑大，尾巴可通过修 reward/数据清理 |
| σ_noise 占 σ_d 大头 | judge 每字段打 3 次取中位数 | 3 倍 judge 调用换 √3 噪声收缩，通常比扩 val 划算 |
| 需分辨的 δ 很小、翻案成本 n 巨大 | 放弃精修，改造 reward 放大分子/缩小分母 | 见 [[reward-design-three-inputs]] 的 reward v2 思路 |

## Claims

### Claim: 配对比较的全部价值是把任务难度这个最大噪声源从比较中剔除——σ_d 显著小于 σ

- **来源**：[[SkillOpt系列04：APO×SkillOpt联合展望——先探索后精修的两段式管道与选型算账方法]]
- **首次出现**：2026-07-15
- **最近更新**：2026-07-15
- **置信度**：0.85
- **状态**：active

> 难题上两个候选**都低**，逐任务相减后"这题难"这个共同因素被抵消，所以 σ_d（video2frames 实测 ≈0.12）显著小于 σ（≈0.20）。且 σ_d 受低分尾影响更小：两个 prompt 往往在同一个半失败任务上一起失败（难度共享），配对相减后尾巴也大幅抵消。关键事实藏在差值列里：单任务差值（±0.03~0.08）比真实平均优势（+0.02）大好几倍——看单个任务分不清是真强还是碰巧，只能靠 n 个任务取平均把不确定度按 σ_d/√n 收缩。评分或 target 模型一换，两个数都要重估。

### Claim: 小于 δ_min 的真实提升在门控眼里与噪声无法区分——accept/reject 退化成抛硬币

- **来源**：[[SkillOpt系列04：APO×SkillOpt联合展望——先探索后精修的两段式管道与选型算账方法]]
- **首次出现**：2026-07-15
- **最近更新**：2026-07-15
- **置信度**：0.85
- **状态**：active

> `δ_min ≈ 2×σ_d/√n`（95% 置信）是 gate 能"看见"的最小真实差异。val=100、σ_d=0.12 时 δ_min≈0.024——比 video2frames 的整个剩余空间（≤0.02）还大，训练就是在烧 token 原地踏步（实战篇 149 万 token 事故的统计学版本）。反解 `n ≈ (1.96×σ_d/δ)²`：想分辨 +0.01 需要约 550 条 val，每步 gate 评估成本完全不成比例。这与 [[automatic-prompt-optimization]] 页"要可靠确认 Δ 需 SE ≤ Δ/2"的教训同源，但把它前移成了**开跑前的例行判断式**而非事后复盘。

### Claim: 三个数全部从已有 eval 产物免费算出——"先算账再开跑"是几分钟算术换百万 token

- **来源**：[[SkillOpt系列04：APO×SkillOpt联合展望——先探索后精修的两段式管道与选型算账方法]]
- **首次出现**：2026-07-15
- **最近更新**：2026-07-15
- **置信度**：0.8
- **状态**：active

> σ_d 来自两列已有分数的对齐相减，δ_min 是一行公式，δ_remain 来自分量饱和度分解——没有任何一个数需要新训练。时点卡在"上一段结束后、下一段开跑前"。video2frames 把账走完：δ_remain（≤0.02）< δ_min（0.024）→ 停，且"停"字一半是任务天花板、另一半是 reward 结构自己造成的（0.4 死权重压缩分子、合并 judge 噪声撑大分母）——算账的价值不止于省钱，还把矛头指回了正确的瓶颈。

## 实践记录

### Practice: 2026-07-15 — video2frames（APO→SkillOpt 两段式管道决策）

- **应用场景**：APO best（+0.018）产出后，决策是否接 SkillOpt 精修
- **实际结果**：成功（判断式给出明确"停"，避免了一次预期无效的训练）
- **偏差与调整**：δ_remain 用了区间估计（0.002~0.02）而非单点；补算了翻案成本 n≈550 作为"停"的第二证据
- **经验教训**：其中大半"问客户的问题"其实是开跑前的测量动作——答案在数据里，不在客户嘴里；reward 改版后三个数全部失效，必须重录 baseline 重走流程
- **置信度影响**：方法在真实任务上闭环一次，Claims 维持 0.8~0.85；管道"翻转后再跑"的预期尚未实测
