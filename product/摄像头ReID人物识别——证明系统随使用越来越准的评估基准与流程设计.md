# 摄像头 ReID 人物识别——证明系统随使用越来越准的评估基准与流程设计

## 背景与约束

面向**摄像头设备厂商**的 AI 人物识别方案：识别人 → 时间总结 → 事件查询。跨摄像头 ReID 准确率是核心指标。方案需在以下真实约束下运行：

- **多场景通用**：厂商不控制部署，摄像头装在哪（办公 / 私人住宅）、怎么装（高度、角度、光照）完全未知，且海量分散——**无法逐点提前校准**。
- **共用一个模型**：模型由开源人脸数据集 + 厂商自标的几百万条数据训练，**人脸已融入 ReID embedding**（衣着无关锚点）。
- **合规红线**：用户上传的图片/视频**仅推理不留存、不得用于训练**；但 **embedding + 用户确认标签可保留**（用于画廊与评估，非训练）。
- **已实现**：多模板画廊 + 聚类取舍（平衡准确率与漂移）。
- **主要痛点**：(b) 误合并（两个相似的人被并成一个身份）、(d) 长期漂移（换衣/换季掉点）。

## 核心方法论

> 「越来越准」在本方案中 = **画廊随主动学习富集 → 匹配变准，模型权重不动**。
> 所以要证明的不是「模型变强」，而是「**同一个模型，画廊越富，越准**」。
> 证明它最干净的工具是「**冻结查询集 × 变化的画廊状态**」做回放。

## 一、两类基准必须分开

| 基准 | 测什么 | 是否随使用变化 | 用途 |
|---|---|---|---|
| **模型基准（静态）** | 模型裸能力 | 否（除非换模型） | 模型升级时的回归门禁 |
| **学习曲线基准（动态）** | 画廊富集带来的增益 | **是——产品要的故事** | 证明「用得越久越准 + 提升多少」 |

- **模型基准**：用厂商几百万自标集 + 公开集，含**换衣专项**（PRCC / LTCC / Celeb-reID，专测痛点 d）。输出 Rank-1/Rank-5/mAP/CMC，作为天花板与回归基线。
- **学习曲线基准**：见下，是给产品/客户的核心物料。

## 二、核心手段：冻结查询回放（stored-embedding replay）

利用「embedding + 用户确认标签 + 时间戳可留存」这一合规事实，构建纵向评估：

1. **留冻结查询集**：每个 deployment 抽一批 `(query embedding, 用户确认的真实身份, 时间戳)`，**只存向量不存图**。
2. **定义画廊知识档位**：G₀（冷启动空画廊）、G₁（每身份 k 次确认后）、…、Gₙ。
3. **同一批冻结 query 依次打到 G₀…Gₙ** → 每档算 Rank-1/mAP → 连成**学习曲线**。
4. 因为 query 固定、只有画廊在变，**增益可干净归因于主动学习**，排除「人群变化 / 用户疲劳」等混淆。

**产出物料**：横轴「每身份确认次数 / 运行天数」，纵轴准确率，一条上升后饱和的曲线，外加「20 次确认后 Rank-1 从 82%→91%」这种可量化结论。

## 三、三层指标（对应真实痛点）

### 算法层（ReID 核心）
- Rank-1 / Rank-5（CMC 曲线）、mAP
- **误合并率（痛点 b）**：身份簇的 purity / 错并率（多少真实身份被错并成一簇）。单独盯——误合并代价远大于漏分。
- **跨衣着 Rank-1（痛点 d）**：只在「跨越换衣事件」的 query 子集上算，看人脸锚点 + 主动学习桥接效果。

### 决策层（阈值方案）
- Precision / Recall / F1 @ 工作阈值
- **DET 曲线（FAR vs FRR）**：阈值方案最该用的图，展示「校准层 + 画廊富集」如何把曲线往左下推。
- 误合并率单列。

### 业务层（端到端，产品/客户关心）
- **找人耗时**：query→用户确认命中的 p50/p90
- **误报率 / 漏报率**
- **用户纠正率随时间下降**：最直观、最便宜的「越来越准」信号（只数纠正次数、不存图），做实时大盘。但有混淆（用户疲劳也会少纠正），故做「活体趋势」，**严谨归因仍靠第二节的冻结回放**。

## 四、把「提升多少」讲成可信结论

- **带置信区间报增益**：对身份/query 做 bootstrap，给「+9pp，95% CI [7,11]」区间，而非单点。
- **饱和点分析**：曲线在「每身份约 N 次确认」后收益递减——回答产品「让用户标多少才够」，管理客户预期。
- **A/B 强归因（可选，最有说服力）**：一组主动学习入库 ON、一组画廊冻结/随机入库，比较轨迹，证明增益**来自主动学习**而非单纯时间流逝。

## 五、两个必须提醒的坑

1. **冷启动地板**：新装摄像头画廊空、准确率低，学习曲线天然从低点爬升。向产品/客户讲清「冷启动 → 爬坡 → 饱和」三段，别让冷启动数据被误读成模型差。
2. **合规复核**：存 embedding + 真实身份标签 + 时间戳做纵向评估，**虽是评估非训练，但仍是生物识别数据留存**。评估管线要可审计地证明「只读不训」，并和法务确认这种「带身份标签的 embedding 长期留存」在住宅场景是否需要额外同意。

## 六、冻结回放的实现：两种泄漏必须同时堵死

学习曲线最容易出的错是**指标虚高**。有两种泄漏：

**泄漏 1：时间泄漏（用未来信息评估过去）**
- 评估 query q（时间戳 t）打到画廊状态 G(t) 时，G(t) **只能包含 t 之前的确认**。
- 实现：所有确认按时间戳排序，`G(t) = {confirmation | timestamp < t}`。档位推进 = 按时间戳顺序回放确认流、在每个档位打快照。

**泄漏 2：query 入廊（自己匹配自己）**
- 若被打分的 query 本身也作为模板进了画廊 → 余弦相似度 1.0 → 假的 100%。
- 实现：**冻结集 Q 与入廊流 E 严格不相交**（Q∩E=∅），从源头杜绝。等价于「入廊 / 评估」split。

## 七、入廊流 E 与冻结集 Q 的切分

```
每个 deployment 的「已确认实例」
  ├── 入廊流 E（带时间戳，喂画廊，复刻生产的聚类/剪枝/衰减逻辑）
  └── 冻结集 Q（held-out，永不入廊，只用于打分）
```

冻结集 Q 的抽样要求：
- **分层覆盖**：高频 + 长尾身份、早期 + 晚期时间、不同光照；**重点纳入「跨换衣事件」的 query** 测痛点 (d)。
- **永不入廊标记**：这些实例即使被用户确认，也打 `eval-only` 标志，绝不进画廊（否则泄漏 2 复活）。
- **规模**：每身份足够做 bootstrap 置信区间。

画廊档位定义：
- **主轴 = 每身份确认次数**（k=0,1,2,5,10,20…）：因果干净，回答「标多少有用」。
- **辅轴 = 运行天数**（day 0/7/30）：贴产品叙事，但被使用率混淆，只做展示。
- 构建 G_ℓ 时**必须套用与生产一致的画廊逻辑**（聚类、剪枝、时间衰减），否则测的是另一个系统。

## 八、覆盖 ≠ 匹配质量（极易误导产品）

低档位时 Rank-1 低，常常**不是匹配差，而是这个人还没进画廊**（coverage 未到）。必须拆成两条曲线：

| 指标 | 口径 | 给谁看 |
|---|---|---|
| **条件 Rank-1** | 只在「身份已入廊」的 query 上算 | 算法诊断——纯匹配质量 |
| **无条件 Rank-1** | 所有 query（含未入廊→记为 miss） | 产品叙事——真实体验提升 |

产品故事用无条件，算法归因用条件。两条一起画，才说得清「提升里多少来自覆盖、多少来自匹配变准」。

## 九、误合并率的精确定义与监控

误合并 = **两个不同真人被分到同一个系统身份**。

**离线金标准（有 ground-truth 标签时）**
- 别只用 cluster purity（小簇刷分、对合并不敏感）。
- 用 **pairwise / B-Cubed precision**：在「被判为同簇」的所有样本对里，真正同人的比例。**误合并率 = 1 − pairwise precision**，直接惩罚合并。
- 配 **pairwise recall（1 − 错分率）** 看权衡。B-Cubed 对簇大小更稳，推荐作主指标。

**在线监控（无完整标签时，用代理信号）**
- **时空矛盾计数（最强、免费、100% 确定）**：系统把**同一帧共现的两个 track** 并进同一身份 → 物理上不可能 → 必是误合并。无需任何标签即可实时统计，最该上大盘。
- **簇内双峰检测**：身份簇的 embedding 分裂成两个分得很开的子簇 → 误合并嫌疑 → 触发人工复核。
- **用户「这不是同一人」纠正**：已确认的误合并。

**监控编排**
- 实时大盘：时空矛盾率（确定）+ 用户合并纠正率 + 簇内双峰告警。
- 周期离线审计：在标注评估集上算 B-Cubed precision / 误合并率作金标准。
- **挂到阈值**：误合并多源于全局阈值偏松；结合 DET 曲线看不同工作点的误合并率。既然不能逐点校准，**query-adaptive 归一化（z-norm/t-norm）+ 时空互斥约束**是主要防线，上述监控量化残余。
