# 论文阅读：SkillOpt — 把 agent skill 当作 frozen agent 的可训练外部状态

> 核心视角：SkillOpt 的贡献不是"发明了文本优化"，而是把 **weight-space SGD 的可复现纪律**（有界学习率、拒绝记忆、动量、validation 门控）第一次完整搬进文本空间，让 agent skill 的优化"像一个真正的深度学习优化器"。本文整合与 [[Agent Lightning系列08：RL实战篇——example选型、calc_x跑通VERL训练与tinker等框架]] 中 APO（自动 prompt 优化）的对比讨论。

---

## 一、它想解决什么问题

摘要开门见山：今天的 agent skill 要么**手写**、要么**一次性生成**、要么通过**松散的自我修订演化**（loosely controlled self-revision）——

> "none of which behaves like a deep-learning optimizer for the skill, and none of which reliably improves over its starting point under feedback."

三种做法都**不像优化器**，也**不保证比起点更好**。SkillOpt 的主张是：**skill 本身应当被"训练"**，用和权重优化同样的纪律。

**为什么不直接训权重？** paper 给了现实约束：

> "weight adaptation is often unavailable for closed frontier models and expensive for open ones, while manually written or one-shot skills are brittle."

闭源前沿模型（GPT-5.5 这类）改不了权重，开源模型训练又贵。于是退而求其次，把"可训练性"转移到一段文本上——**skill document 是"训不动权重时的权重替身"**。

**形式化（Eq.1）**：对 harness `h`、任务 `x`、skill `s`，执行产生轨迹 `τ(s)` 和标量分数 `r(s)`。skill `s` 是"a natural-language policy **inserted into the agent context before execution**"，target model `M` **冻结**，optimizer `O` 独立、离线。最终产物是一份紧凑、可"训练/验证/导出/复用"的 **`best_skill.md`（300~2000 tokens）**，部署时**零额外模型调用**。

## 二、四大机件——把 SGD 纪律搬进文本空间

这是全文的核心。SkillOpt 逐一对标深度学习优化器的关键组件：

| 权重空间（SGD） | SkillOpt（文本空间） | 归属 |
|---|---|---|
| 参数 θ | skill document（frozen agent 的外部状态） | 成品 |
| 一次参数更新（方向） | 单条 edit（add/delete/replace/insert 原子补丁） | — |
| **learning rate / 步长** | **textual LR budget `L_t`（每步最多几条编辑）** | — |
| **momentum / 负样本记忆** | **rejected-edit buffer（记住失败编辑）** | 优化器侧 |
| **slow weights / EMA / 双时间尺度** | **slow-update field（受保护慢字段）** | 成品内 |
| **meta-learning / L2O** | **meta skill `m_meta`（编辑心法）** | 优化器侧，不出厂 |
| validation early-stopping | **held-out validation gate（严格改进才接受）** | — |

### 2.1 textual learning rate `L_t` —— 用"编辑条数"离散代理步长

文本空间**没有连续度量、没有梯度**，"语义步长"无法像权重那样连续量化。SkillOpt 的做法是**回避这个问题、用离散代理**：

> "the edit budget `L_t`: the maximum number of skill edits applied at step t"

即 **LR = 每步允许的编辑条数**，不是 token 数、不是 edit distance、更不是语义距离。默认 `L_t=4`，**cosine 衰减**到下限 2，实验扫 `{1,2,4,8,16}`。它约束的是"**同时改几处**"，不约束"每处改多深"——语义探索发生在**单条编辑内部**，`L_t` 管的是**并发编辑数量**。

**消融数据（关键，纠正"越小越好"的直觉）**：one-factor ablation 显示 `L_t` 的**具体取值鲁棒、benchmark 相关且带噪声**——`L_t=1` 完全有竞争力（SearchQA 85.5 / Spreadsheet 77.5），LiveMath 上 `L_t=1`（62.1）甚至反超 `L_t=4`（56.5）。paper 定性为 **"small or moderate edit budgets are competitive throughout"**。

| `L_t` | SearchQA | SpreadsheetBench | LiveMath |
|----|----|----|----|
| 1 | 85.5 | 77.5 | 62.1 |
| 4（默认） | 86.5 | 78.2 | 56.5 |
| 8 | 87.0 | 73.6 | 66.9 |
| without lr（无界改写） | 84.6 | 75.7 | 57.3 |

**真正的分水岭是"有界 vs 无界"，不是 4 vs 1**：任何 moderate 有界预算都在 85~87，无界改写掉到 84.6。所以 paper 的论点是 **"要有界、且界得适中"，不是"界得越紧越好"**——`"each revision remains close enough to the last one"` 里的 **"close enough" ≠ "smallest possible"**。

**那为什么默认 4 而非 1？**（paper 未声称 4 更优，是务实中值）
1. **成本**：每步 = 一次 rollout + 一次 held-out 验证；`L_t=1` 打同样多编辑需约 4 倍 step，评测开销 4 倍。
2. **协同编辑（`L_t=1` 的真实短板）**：验证门**对整份候选打包接受/拒绝**（无 per-edit 验证），有些改动**只有成组才涨分**（如"加新规则 + 删冲突旧规则"）。`L_t=1` 提不出联合 move，会卡在"单条都不涨、成组才涨"的局部最优；`L_t>1` 才能一步跨谷。
3. **explore-then-refine**：cosine `4→2` 早期大步探索、后期小步精修；但 panel(e) 显示 constant/cosine/linear 差别不大，**"the bounded-update story does not depend on a single specific scheduler"**。

### 2.2 rejected-edit buffer —— 负经验记忆（epoch 内）

> "The optimizer records an epoch-local buffer containing observed failure patterns and, for rejected steps, the edits that were tried and the score drop they caused."

候选 skill 没过 validation gate 时，它提出的编辑被追加进 buffer，**同 epoch 内**的后续 reflection 会拿到它作为负信号：避免重复失败编辑、聚焦未解决的失败。**epoch 边界清空**——因为换 epoch 后样本重排、skill 也变了，旧失败模式可能不再成立，故意让它短命。

### 2.3 slow-update field —— 慢权重 / 双时间尺度（成品内）

epoch 末做一次"回测 diff"：

> "At the end of an epoch, SkillOpt samples the same training items under the previous epoch's skill and the current skill, then groups them into improvements, regressions, persistent failures, and stable successes."

把纵向总结写进 skill 文档里一块 **markup 围栏的 protected slow-update field**，**step 级快编辑改不动它**：

> "Step-level edits cannot overwrite the protected slow-update field, so fast local changes and slower epoch-wise consolidation remain separated."

它随成品部署，作用是**沉淀长期有效规则、防止被快编辑冲掉**。从其 system prompt（`slow_update.md`）看，它的角色是 **"strategic skill advisor"，直接对训练模型说话**（"When you encounter X, always do Y"），优先级为 **(1) 防回退 (2) 修持续失败 (3) 强化成功模式**，且**不与主 skill body 重复、只互补**。

### 2.4 meta skill `m_meta` —— "改 skill 的 skill"（优化器侧，不出厂）

它 summarize 的是**"编辑模式本身"**（哪类改法有效/无效），epoch 末更新，prepend 到**未来 optimizer 的 prompt**，**never shipped with the deployed artifact**。类比 meta-learning / learning-to-optimize：**不是学任务，是学"怎么改 skill"**。从其 system prompt（`meta_skill.md`）看，角色是 **"optimizer coach"，直接对未来的优化器说话（"not the training model"）**，改进优化器的 failure analysis / success analysis / patch merging / edit ranking 各阶段——即"哪类编辑有效、什么抽象层级合适、哪些 regression risk 要防"。

⚠️ **注意**：`m_meta` 修改的是 optimizer 的 **context（prompt），不是权重**——optimizer 模型本身也冻结。所以它是给优化器一张 run 内的草稿纸，不是"训出一个更强的优化器"。

### 2.3+2.4 辨析：slow_update vs meta_skill —— 本质是"讲给谁听"

两者读**同一场纵向复盘**（同批任务在新旧 skill 下的四分类对比）、都在 epoch 末写、都**自我修订**（反思上一版，retain 有效/remove 无效），但 audience 相反：

| | slow_update.md | meta_skill.md |
|---|---|---|
| 说话对象 | **训练模型** | **未来的优化器**（"not the training model"） |
| 写什么 | 任务怎么做 | 编辑怎么写 |
| 输出去向 | skill 的 protected section | 优化器侧内存 |
| 出厂 | **是** | **否** |
| 层级 | object-level（策略） | meta-level（learning-to-optimize） |

**类比**：slow_update = 教练贴在更衣室的**战术板**（给运动员、比赛带着走）；meta_skill = 教练兜里的**执教笔记**（给自己、不给运动员看）。

**精修（纠正"累积"提法）**：二者都不是 append-only 日志，而是**每 epoch 覆盖重写 + 保留仍有效部分**（prompt：`"OVERWRITE the previous guidance"` 且 `"retains and strengthens parts that proved effective"`）——即**策展式滚动记忆（curated running memory）**，靠这个保持 compact / high-signal。

**结构揭示**：优化器内部其实是三个角色——`per-step analyst`（快，local patch）+ `strategic skill advisor`（slow_update，对外）+ `optimizer coach`（meta_skill，对内）。一快两慢，两慢一对外一对内。

### 2.5 三时间尺度 —— epoch / run / step 的作用域区分

```
run（一次完整优化，产出一个 best_skill.md）
 └── epoch（一遍训练集）×N
      └── step（一个 minibatch reflection，打 ≤L_t 条补丁）×M
```

| 记忆 | 更新频率 | 生命周期（作用域） | ships? |
|---|---|---|---|
| rejected-edit buffer | 每 **step** 追加 | **单个 epoch**（epoch 末清空） | 否 |
| slow-update field | **每 epoch 末覆盖重写**（保留有效部分） | **整个 run** | 是 |
| meta skill `m_meta` | **每 epoch 末覆盖重写**（保留有效部分） | **整个 run** | 否 |

关键：只有 **buffer 是 epoch-scoped**（短命）；slow-field 和 `m_meta` 是"**epoch 触发更新、run 存活**"的 run-scoped 长记忆。三者天花板都是 **run**——**没有一个跨到"另一个 skill 的 run"**（详见第五节的作用域讨论）。

### 2.6 三个 split 与"诊断 vs 接受门"辨析

一个易混点：**epoch 末 slow/meta 复盘用的数据**，和 **step 级接受候选 skill 用的验证门**，是不是同一个 split？答案是**不同 split、不同角色**。

**三个数据划分（Eq.2）**：

| split | 记号 | 作用 |
|---|---|---|
| 训练集 | `D_tr` | 跑 rollout、暴露失败/成功样本、供 optimizer 反射提编辑 |
| 选择集（held-out） | `D_sel` | **接受门**：候选 skill 在此打分，严格改进才被接受 |
| 测试集 | `D_test` | 最终评测，不参与任何优化决策 |

**两个用途，别混**：

| | step 级接受门 | epoch 末纵向复盘 |
|---|---|---|
| 用哪个 split | **`D_sel`（held-out）** | **`D_tr`（重采样约 20 条训练项）** |
| 性质 | **裁决**（accept / reject） | **诊断**（四分类：improvements / regressions / persistent failures / stable successes） |
| 输出 | 收/不收这份候选 | 一段供 slow/meta 参考的纵向总结 |
| 粒度 | bundle 级（整份候选一起评） | 逐样本比新旧 skill |

**关键澄清（纠正此前的猜测）**：epoch 末 slow/meta 写入的那份"带 slow-field 的候选 skill"，**本身仍要过 `D_sel` 接受门**——`"this candidate is still passed through the validation gate"`。也就是说 `D_tr` 复盘只负责**产生素材**（诊断"哪些回退了、哪些持续失败"），素材写进 slow-field 后形成的新候选，**照样要在 `D_sel` 上证明自己严格改进才落地**。slow/meta 不是"绕过验证的特权写入"，而是**换了个信息来源（纵向诊断）去 propose，验证纪律不变**。

### 2.7 快慢两层循环 —— 慢层读什么、怎么进一步优化

把 2.5/2.6 合起来看，SkillOpt 就是**一快一慢两层嵌套循环**：

```
epoch（慢层）
 └── step（快层）× M：每步 propose ≤L_t 条编辑 → D_sel 接受门 → 收/退
 └── epoch 末（慢层）：D_tr 前后纵向诊断 → 写 slow-field / meta → 仍过 D_sel 接受门
```

**两个必须拧准的点（否则会误解慢层）**：

**① 慢层读的是"快层的净结果"，不是"快层的编辑过程"。** 慢层不去翻这一 epoch 内那一条条 step 编辑记录，而是拿 **上一版 skill vs 当前版 skill**，在 `D_tr` 重采样上各跑一遍，比对**行为差**。类比：不是复盘"这周改了哪 40 处代码"，而是"上周的我 vs 这周的我，在同一批题上哪些变好、哪些变差"。

**② gate 在两层都有，不是只有慢层才校验。** 快层每个 step 的候选本来就在过 `D_sel` 接受门；慢层诊断产出的 slow-field 候选**同样过 `D_sel`**。慢层不是"补一道校验"，而是"换个信息源再 propose 一次，纪律照旧"。

**慢层拿到四分类诊断后，怎么"进一步优化"——每一类触发一种不同动作**（对应 slow_update 优先级：防回退 > 修持续失败 > 强化成功）：

| 诊断类别 | 慢层优化动作 | 为什么归慢层 |
|---|---|---|
| **regressions**（新版反而变差） | 写受保护 guard 规则抵消 | 回退多是快层为讨好某 minibatch 过拟合的漂移；只有慢层用更广 `D_tr` 才照得出 |
| **persistent failures**（新旧都错） | 下策略级新指导 | 快层单步小改陷局部最优，慢层能提"换打法"的高层 move |
| **improvements**（新版变好） | 归纳"为什么好"固化成规则 | 把偶然涨分变成可复用显式规则，防下 epoch 被冲掉 |
| **stable successes**（一直对） | 锁住、标记受保护 | 避免后续编辑误伤 |

**"进一步优化"的两条实际杠杆**：

1. **受保护区 = 防漂移棘轮**：这些规则写进 step 快编辑**改不动**的 protected slow-field，给快层探索加一道"只进不退"的棘轮——快层继续大胆试错，慢层沉淀的长期规则不被高频小改磨掉。这正是**双时间尺度 / 慢权重**的本体：快权重探索、慢权重巩固。
2. **meta_skill 优化"优化器本身"**：同一份诊断，meta 侧读出的是**编辑模式**教训（哪类编辑总引发 regression、什么抽象层级有效、哪些 patch 合并有风险），prepend 回优化器 prompt，改进**下一轮 propose 的质量**。

一句话：**慢层不是"再改一版 skill"，而是用更广视野把快层的漂移纠回来（防回退）+ 啃快层啃不动的（修持续失败）+ 把偶然固化成必然（强化成功），并顺带教优化器下次别再犯同类编辑错误。**

## 三、训练循环（算法顺序）

1. 冻结 target model `M` 用当前 skill 在训练集上跑 rollout batch
2. optimizer 分离失败/成功样本，划成 minibatch
3. optimizer 提出 **failure-driven**（改错）+ **success-preserving**（保对）的 add/delete/replace 编辑
4. 编辑分层合并（**失败优先**），按 utility 排序
5. 应用 top `L_t` 条编辑（有界预算 + cosine 衰减）
6. 候选 skill 在 **held-out selection split（`D_sel`）** 上评测
7. **接受门控：必须严格改进（ties rejected）**
8. 被拒编辑存入 epoch-local buffer 作负反馈
9. epoch 末：在 **`D_tr` 重采样**上对比上一版与当前 skill，做四分类纵向**诊断**
10. optimizer 据诊断写纵向指导进 protected slow-update field；**该候选仍过一次 `D_sel` 接受门**才落地
11. 总结 `m_meta`（仅优化器侧）prepend 到后续 reflection prompt

## 四、与竞品的本质区别——"缺哪块"式对比

paper 把 baseline 分三族：**one-shot**（human skill / one-shot LLM skill）、**prompt optimization**（TextGrad / GEPA）、**skill evolution**（Trace2Skill / EvoSkill）。用**两个正交维度**看得最清：

维度 A：优化对象是不是**持久可复用的 skill artifact**（vs 一次性 prompt）
维度 B：有没有**优化器纪律**（validation gate / bounded LR / rejected-edit buffer / slow-meta）

| 方法 | 优化对象 | validation gate | bounded LR | rejected buffer | slow/meta |
|---|---|:---:|:---:|:---:|:---:|
| human / one-shot LLM | 静态 skill，不更新 | — | — | — | — |
| **TextGrad** | prompt（非持久，一次性精修） | ✗ | ✗ | ✗ | ✗ |
| **GEPA** | prompt（Pareto 反射演化，transient） | ✗ | ✗ | ✗ | ✗ |
| **Trace2Skill** | skill artifact（轨迹蒸馏） | **✗（关键缺）** | ✗ | ✗ | ✗ |
| **EvoSkill** | skill folder（多候选演化） | 部分（失败分析） | **✗** | **✗** | ✗ |
| **SkillOpt** | 单一 skill document | ✓ | ✓ | ✓ | ✓ |

- **TextGrad**："gradient-style 自然语言 prompt 优化"，但 **"optimize prompts but not a persistent skill artifact"**——停在一次性 prompt 层。
- **GEPA**："Pareto reflective prompt evolution"，**"mainly target prompts, system designs, or full configurations rather than reusable domain adaptation"**——把 prompt 当临时目标而非可审计持久 artifact。
- **Trace2Skill**：轨迹级 skill 蒸馏，**已跨过"产出持久 skill"这道坎**，但 **"lacks SkillOpt's iterative validation gate"**——蒸馏结果没 held-out 验证就部署，就差那道泛化守卫。
- **EvoSkill**：skill-folder 演化，**"lack both bounded textual learning rates and rejected-edit memory"**——无步长控制（编辑无界累积）、无负反馈记忆（重复踩坑、震荡）。

**总纲**：四个竞品分别缺"持久 artifact"（TextGrad/GEPA）、"validation gate"（Trace2Skill）、"步长控制+负反馈记忆"（EvoSkill）。**SkillOpt 不是某一点更强，而是把这四块首次凑齐——没有短板**，这解释了它为什么 52 个 cell 全胜。

> 👉 **TextGrad + GEPA 正是 APO（自动 prompt 优化）家族**。[[Agent Lightning系列08：RL实战篇——example选型、calc_x跑通VERL训练与tinker等框架]] 里讨论的 agent-lightning APO 就落在这一档：prompt 优化、transient、无 SkillOpt 那套纪律。

## 五、几个深层讨论点

### 5.1 skill document vs prompt —— 本体同一，区别在定位与纪律

**在优化循环层面，skill 和 prompt 是同一件事**：都是"template 作变量 → LLM 探索 → rollout → 打分 → LLM 挑优化版"。skill 的定义就是"执行前塞进 agent context 的自然语言"，机制上就是 prompt 内容。

区别是**定位**：prompt 偏任务框定（输入侧）；skill document 被刻意框成 **frozen agent 的可训练外部状态 `s`**，类比权重 θ——一个持久、可"训练/验证/导出/复用"的 artifact。**定位不同才带来那套纪律，本体确实同一。**

### 5.2 权重级 vs 文本级 tool-use 优化（对比 agent-lightning）

SkillOpt **只修改文本**（`best_skill.md`），不碰权重、不改 tool schema / MCP / harness。但它**优化的内容可以是工具使用策略**——在 Codex（+24.8）、Claude Code（+19.1）这类带工具调用的 agentic harness 里评测，增益证明它优化了 tool-augmented 行为，只是**把 tool-use 策略写成文本 playbook，而非烧进权重**。

| | agent-lightning SFT/RL | SkillOpt |
|---|---|---|
| 优化对象 | **policy / 权重 θ** | **context / 外部状态（文本）** |
| tool-use 怎么被优化 | 信号流经真实 tool-call 动作/token，参数层学会 | 把流程写成 procedure，冻结模型去读 |
| 能否教会新能力 | 能（权重层装入原本不会的技能） | 不能（只能编排已有能力，受底座天花板限制） |
| 模型要求 | 可训练（开源/有权重访问） | **冻结**（专为闭源前沿模型设计） |
| 迁移性 | 权重绑单一模型，换模型失效 | 文本与模型/harness 无关，**可迁移** |

**能力边界**：RL 能在权重层装新工具能力；SkillOpt 装不了——冻结模型根本调不动的工具，再好的 skill 文本也救不回来。**两者是互补层，可叠加**：先 RL/SFT 练强原生工具能力，再用 SkillOpt 给（可能闭源的）agent 配一份可迁移 playbook。

### 5.3 经验的作用域——它怎么"判断"对哪些 skill 有效？

**答案很优雅：它不预先判断，用 validation gate 事后筛。** 无论提案来自 buffer / slow-field / `m_meta`，这些经验**只负责 propose 编辑**，而**这批编辑打包成的候选 skill 必须过当前 skill 的 held-out validation gate 才被接受**。注意 gate 是 **bundle 级**（一次评整份候选 `s̃`，`if score_cand > score_cur` accept，**无 per-edit 验证**），不是逐条评。所以"对当前 skill 是否有效"由 gate 用当前任务分数**当场、empirical** 裁决，不需要给经验贴"通用/部分/局部"标签。这也是它比 EvoSkill（无 gate、经验无差别累积）稳的原因——也正因为 gate 是 bundle 级，`L_t>1` 才能让"只有成组才涨分"的协同编辑一次通过（见 2.1）。

**跨 skill 是否有效？** 由 **transfer 实验**回答，且迁移的是**成品 artifact**（不是优化器元经验），强调 **nearby**（相近域）、效果是**测出来**而非假设的。**当前 `m_meta` 不跨 run 复用**——"reuse of optimizer-side meta skills across benchmarks"被明确列为**未来工作**，印证了这一局限。

### 5.4 validation 信号的本质——选择门 vs 梯度（对比 agent-lightning reward）

一个常见对比：SkillOpt 用 **ground truth** 做 validation（SearchQA 的 `hard` 就是模型答案对参考答案的可验证分），而 agent-lightning 用 **reward**——这算不算两者的大区别？**答案：ground-truth ↔ judge 这条轴是正交的、可插拔的，不构成框架级分野；真正的区别在"分数拿去干什么"。**

**先纠正前提——reward 来源两边都能换**：

- agent-lightning **不是"主要用 judge"**，它 reward-agnostic：calc_x 实战用的恰是 **ground-truth 可验证 reward**（算对没有），没接 judge。judge 只是它*支持*的一种来源。
- SkillOpt 也**不限于 ground truth**：gate 吃的是"评估器返回的分数"，默认配 verifier 最好，但论文把 **preference-driven / reward-free validation gate**（用 judge/偏好）明确列为未来工作；`soft` 分在部分 benchmark 就可以是软性/语义评估。

所以 verifier ↔ judge 是**评分手段**的选择，与框架无关。

**本质区别——这个分数驱动的是"选择"还是"梯度"**：

| | SkillOpt | agent-lightning RL |
|---|---|---|
| 分数角色 | **选择 / 门控信号** | **梯度信号** |
| 拿去做什么 | accept/reject 一份候选 skill（改文本） | 算 advantage → PPO/GRPO 更新权重 |
| 类比 DL | model selection / early stopping | 反向传播优化 |
| 在哪测 | **held-out `D_sel`**（防对训练 minibatch 过拟合） | 通常直接在 rollout batch 上算 |
| 粒度/方向 | bundle 级标量比较"是否严格超过当前"（ties rejected） | 连续标量喂 advantage，可稠密可 shaped |

一句话：**SkillOpt 把分数当"筛子"（留不留这次编辑），agent-lightning RL 把 reward 当"梯度"（往哪个方向挪权重）。** 前者离散 accept/reject，后者连续参数更新——这才是真正的两条路。

**派生的次级差异**：因为用途不同，SkillOpt 对分数的绝对校准要求更宽松（只要能可靠区分 A>B 即可）；RL 的 reward 直接进梯度、还要估 advantage，对稠密性与噪声更敏感。这是"用途不同"的下游结果，而非根因。

> 这条也呼应本 vault 的拒绝采样谱系：SkillOpt 的 validation-gated 接受与 [[paper/2026-06-28-RAFT-Reward-rAnked-FineTuning-论文解读|RAFT（拒绝采样 SFT）]] 同源——都是"打分筛选、只留改进"（选择族），区别只是 RAFT 筛**样本**、SkillOpt 筛**编辑**；而"打分做梯度"的 RL 是另一族。

## 六、实验设置与结果

- **6 benchmark**：SearchQA、SpreadsheetBench、OfficeQA、DocVQA、LiveMathematicianBench、ALFWorld
- **7 target model**：GPT-5.5 / 5.4 / 5.4-mini / 5.4-nano / 5.2、Qwen3.5-4B、Qwen3.6-35B-A3B
- **3 harness**：direct chat（system prompt 注入）、Codex（workspace sandbox）、Claude Code（workspace contract）

**Headline（GPT-5.5，相对 no-skill baseline）**：direct chat **+23.5**、Codex **+24.8**、Claude Code **+19.1**。**52/52 cells best 或 tied-best**；对 per-cell oracle（best-of-six 竞品）仍 **+5.4**（82.3 vs 76.9）。

**Transfer**：
- **跨模型**：SpreadsheetBench（GPT-5.4 源）→ mini +9.4、nano +3.0，四行全正
- **跨 harness**：SpreadsheetBench Codex→Claude Code **+59.7**、Claude Code→Codex **+43.6**（迁移增益极大）
- **跨 benchmark**：OlympiadBench→Omni-MATH 正迁移（GPT-5.4 +3.7、mini +1.8、nano +1.3）

## 七、局限与展望

**局限**：
- 需要**打分轨迹 + held-out selection split**，配自动 verifier 效果最好；主观成功的开放域需要更强评估
- 训练成本**只有 skill 被复用时才摊得平**
- **单一 skill 对高度异构的域可能不够**
- 优化后的 skill 编码了**训练分布的启发式**，迁移前需谨慎评估

**展望**（原文列的自然延伸）：跨域共享基础设施的 **skill 库**、**跨 benchmark 复用优化器侧 meta skill**、开放任务的 **reward-free / preference-driven validation gate**、把优化后的 skill **self-distill 回 target model**（这条正好和 agent-lightning 的 SFT/蒸馏路线打通）。
