# Caveman 与 RTK 对比——两种互补的 LLM Token 优化方案

> AI Coding Agent 的 token 膨胀来自三个方向：LLM 自身输出的冗长回复、shell 命令返回的大量噪音、以及每次会话加载的记忆文件。Caveman 和 RTK 分别从不同维度切入这个问题——前者是 Prompt Engineering，后者是数据管道过滤。本文系统对比两者的机制、风险与适用边界，并探讨如何让它们协同工作实现三层 token 全面优化。

---

## 一、问题背景：为什么 Token 优化是 AI Coding 的关键议题

AI 编程助手（Claude Code、Codex、Cursor 等）在一次典型的 30 分钟开发会话中，token 消耗来自三个方向：

| 来源 | 典型消耗 | 示例 |
|------|---------|------|
| **LLM 输出**（回复） | 数千 token/轮 | 解释 Bug、架构建议、代码审查 |
| **工具输出**（命令结果） | ~118,000 token/会话 | git status、cat、grep、test 输出 |
| **记忆文件**（上下文加载） | 数百~数千 token/会话 | CLAUDE.md、项目配置文件 |

token 膨胀带来双重压力：
- **成本**：按 token 计费的 API 调用直接影响开发预算
- **上下文窗口**：过多噪音挤占有效信息的位置，可能导致模型"遗忘"关键上下文

Caveman 和 RTK 正是针对这三个方向中的不同环节提出的解决方案。

---

## 二、Caveman 核心机制回顾

> 详细分析见 [Caveman 深度解析——LLM Token 压缩的 Prompt Engineering 之道](Caveman深度解析——LLM%20Token压缩的Prompt%20Engineering之道.md)

[Caveman](https://github.com/JuliusBrussee/caveman) 是一个面向 40+ AI 编程代理的对话压缩技能插件，其 token 节省分为两层：

**输出压缩**（核心）：通过 SessionStart Hook 将风格约束规则注入系统提示，让 LLM 在同一次推理中直接生成简洁回复——删除冠词、填充词、客套话，使用片段句和短同义词。不存在额外 LLM 调用或输出后处理。平均节省 65%（范围 22%–87%）。

**输入压缩**（caveman-compress）：本地 Python 脚本构造压缩 Prompt 并调用 Claude API，将 CLAUDE.md 等记忆文件重写为精简格式。一次压缩，多次受益。平均节省 46%。

关键特征：
- 输出压缩**零额外 API 调用**，利用 LLM 自身的指令遵循能力
- 六档压缩级别（Lite / Full / Ultra / Wenyan-*）
- Auto-Clarity 安全阀在高风险场景自动恢复详细输出
- 规则自身占 552 tokens 系统上下文

## 三、RTK 核心机制

[RTK（Rust Token Killer）](https://github.com/nicholasgasior/rtk) 是一个用 Rust 编写的高性能 CLI 代理，定位于 AI 编程助手与操作系统 shell 之间的中间层。

### 3.1 架构定位

RTK 的核心功能：**在命令输出到达 LLM 上下文之前进行过滤和压缩。** 以单一 Rust 二进制文件形式提供，零运行时依赖，每条命令处理开销不到 10 毫秒。

### 3.2 工作流对比

```
没有 RTK：
Claude  --git status-->  shell  -->  git
  ^                                   |
  |        ~2,000 tokens（原始）       |
  +-----------------------------------+

使用 RTK：
Claude  --git status-->  RTK  -->  git
  ^                      |          |
  |   ~200 tokens        | 过滤     |
  +------- （已过滤）-----+----------+
```

### 3.3 四大过滤策略

| 策略 | 作用 | 示例 |
|------|------|------|
| **智能过滤** | 去除噪音（注释、空白、样板代码） | 删除 git status 中的提示文字 |
| **分组** | 聚合相似项 | 按目录分文件，按类型分错误 |
| **截断** | 保留相关上下文，删除冗余 | test 输出只保留失败用例 |
| **去重** | 合并重复日志行并计数 | "ERROR: timeout" ×47 |

### 3.4 PreToolUse Hook 机制

RTK 通过 PreToolUse Hook 透明拦截 AI 代理发出的 Bash 命令。例如将 `git status` 自动改写为 `rtk git status`。AI 代理完全不感知这个改写过程，只接收到压缩后的输出。

### 3.5 基准数据

以下为 RTK 官方给出的 30 分钟 Claude Code 会话典型 token 节省估算：

| 操作 | 频率 | 标准输出 tokens | RTK 输出 tokens | 节省 |
|------|------|---------------|----------------|------|
| ls / tree | 10x | 2,000 | 400 | 80% |
| cat / read | 20x | 40,000 | 12,000 | 70% |
| grep / rg | 8x | 16,000 | 3,200 | 80% |
| git status | 10x | 3,000 | 600 | 80% |
| git diff | 5x | 10,000 | 2,500 | 75% |
| git log | 5x | 2,500 | 500 | 80% |
| git add/commit/push | 8x | 1,600 | 120 | 92% |
| cargo test / npm test | 5x | 25,000 | 2,500 | 90% |
| ruff check | 3x | 3,000 | 600 | 80% |
| pytest | 4x | 8,000 | 800 | 90% |
| go test | 3x | 6,000 | 600 | 90% |
| docker ps | 3x | 900 | 180 | 80% |
| **总计** | | **~118,000** | **~23,900** | **~80%** |

### 3.6 Tee 模式兜底

当命令执行失败时，RTK 的 tee 模式会保存完整未过滤的原始输出，以便 LLM 在需要时读取详细错误信息而不必重新执行：

```toml
[tee]
enabled = true          # 默认开启
mode = "failures"       # 可选 "failures"、"always"、"never"
```

## 四、系统性对比

### 4.1 机制层对比

| 维度 | Caveman | RTK |
|------|---------|-----|
| **作用层面** | LLM 对话输出 + 持久记忆文件输入 | 外部工具/命令的输出（shell stdout/stderr） |
| **核心方法** | 风格约束规则注入 + 本地脚本文件重写 | 确定性算法过滤（过滤/分组/截断/去重） |
| **触发方式** | LLM 在生成回答时遵循注入的规则 | PreToolUse Hook 拦截命令并路由至 RTK |
| **处理位置** | 主 LLM 推理过程内 / 本地 Python 脚本 | 本地 Rust 二进制，在 LLM 外部执行 |
| **典型节省** | 输出平均 65%；输入平均 46% | 综合约 80%（范围 70%–92%） |
| **运行时开销** | 零（输出压缩）/ Python + API（文件压缩） | <10ms 延迟 |
| **支持平台** | 40+ AI 代理 | Claude Code、Copilot、Cursor、Gemini CLI、Codex |

### 4.2 语义保真度对比

**Caveman** 在语义层面进行重构：删除赘述和语气词，用更短的措辞表达同一含义。

- 保护区域：代码块、URL、文件路径、命令、标题、日期、版本号严格原样保留
- Auto-Clarity 安全阀在高风险场景自动恢复
- 官方基准声称维持 100% 技术准确率

**RTK** 的语义修改风险更低：不重写内容文字，仅执行结构性删减和重组（去空白、合并重复、按类型分组）。

- Tee 模式在命令失败时保存完整原始输出
- 主要风险在于过滤规则偶有误判，可通过 `exclude_commands` 配置排除

**独立测试揭示的真实效果**：在已有简洁指令的基线条件下，Caveman 实际节省仅 13%–21%，但全部测试中无数据丢失。RTK 基于确定性算法，效果稳定可预测。

### 4.3 大模型调用与成本

| 维度 | Caveman | RTK |
|------|---------|-----|
| 是否调用额外 LLM | 否（输出压缩）；是（文件压缩调用 Claude API） | 否，纯本地 Rust 程序 |
| 自身的 token 开销 | 完整技能规则约 552 tokens（注入到上下文中） | 零（不占用 LLM 上下文） |
| 净收益权衡 | 552 tokens 开销 vs 平均 65% 输出节省 | 无 token 开销，唯一成本是 <10ms 延迟 |

关键推论：当 LLM 输出本来就很简洁时（如结构化数据提取任务），Caveman 的 552 tokens 注入开销可能抵消部分节省。此时用 85 tokens 的精简版规则更划算。

### 4.4 调试与可观测性

| 维度 | Caveman | RTK |
|------|---------|-----|
| 模式开关 | `/caveman off` 或说 "normal mode" 即时关闭 | 通过 `exclude_commands` 排除特定命令 |
| 压缩级别调节 | 支持 `/caveman lite/full/ultra` 动态切换 | 不适用（按命令类型自动选择过滤策略） |
| 原始内容保留 | caveman-compress 备份 `.original.md` 文件 | Tee 模式在命令失败时保存完整输出 |
| 安全机制 | Auto-Clarity 自动退出简化模式 | Graceful degradation：不支持的命令直接透传 |
| 统计功能 | 状态栏 badge 显示模式状态 | `rtk gain` 显示 token 节省统计；`rtk gain --graph` 30 天 ASCII 图表 |

### 4.5 跨会话机制对比

| 维度 | Caveman | RTK |
|------|---------|-----|
| 新会话是否需要操作 | 有 Hook 的平台无需操作；否则需手动 `/caveman` | 无需任何操作 |
| 状态持久化方式 | `.caveman-active` 文件 + 压缩后的记忆文件 | `~/.config/rtk/config.toml` |
| 上下文一致性风险 | LLM 可能在超长对话中指令遵循衰减 | 无（确定性算法，每命令独立） |
| 更新机制 | 修改 `.original.md` → 重新运行 compress | 修改 `config.toml` → 即时生效 |

## 五、风险与不适用场景

### 5.1 Caveman 的风险

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **解释不足** | Ultra 模式省略推理过程和前提条件 | Auto-Clarity 部分缓解；对新领域慎用 |
| **基线效果递减** | 当已有简洁指令时，边际收益 13%–21% | 使用精简版规则（85 tokens） |
| **规则消耗 token** | 完整技能占 552 tokens | 短对话中占比不容忽视 |
| **多轮漂移** | LLM 指令遵循在超长对话中可能衰减 | Per-turn reinforcement 缓解 |
| **不可量化保证** | 同一问题在不同模型/上下文下压缩效果不同 | 接受 22%–87% 的内在变异性 |

### 5.2 RTK 的风险

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **过滤误伤** | 非标准格式的命令输出可能误删有用信息 | `exclude_commands` 手动排除 |
| **非文本不适用** | 不处理二进制数据、图片等内容 | 仅用于文本输出的命令 |
| **覆盖范围有限** | 支持 100+ 命令，超出范围直接透传 | 随版本更新扩展 |
| **纯对话无效** | 不涉及 shell 命令执行时无法发挥作用 | 纯问答场景用 Caveman |

## 六、选择决策树

### 6.1 场景矩阵

| 使用场景 | Caveman | RTK | 建议 |
|---------|---------|-----|------|
| 代码问答 / Bug 分析 | 高效 | 无效（无命令输出） | 用 Caveman |
| AI 驱动的构建/测试执行 | 有限帮助 | 高效（90% 压缩） | 用 RTK |
| AI Agent 频繁执行 shell 命令 | 有限帮助 | 核心场景 | 用 RTK |
| 长期项目、频繁新建会话 | caveman-compress 持续节省 | 全局 Hook 持续节省 | **两者结合** |
| PR 审查 / Commit 生成 | 有专门子技能 | 有限帮助 | 用 Caveman |
| 教学 / 文档撰写 | 不适合 | 不相关 | 关闭 Caveman |
| 中文开发环境 | 文言文模式加强 | 提供中文文档 | 两者可用 |

### 6.2 决策流程图

```
你的 AI Coding 工作流是什么？
  │
  ├─ 主要是问答/代码审查（少命令执行）
  │   └─ → 用 Caveman（选 Full 或 Lite）
  │
  ├─ 主要是命令驱动（频繁 git/test/build）
  │   └─ → 用 RTK
  │
  ├─ 两者兼有（问答 + 命令执行）
  │   └─ → 两者同时启用
  │
  └─ 需要深度推理（复杂架构/算法/多步逻辑）
      └─ → 都不用（或仅用 RTK 压缩命令输出）
```

## 七、联合使用——三层 Token 全面优化

### 7.1 互补性分析

Caveman 和 RTK 作用于**完全不同的数据流**，可以同时启用而互不干扰：

```
                    Token 优化三层架构

  ┌────────────────────────────────────────────────────┐
  │  Layer 3: 输出层（LLM 回复）                        │
  │  优化工具: Caveman 输出压缩                          │
  │  机制: 系统提示约束 → LLM 直接生成精简回复             │
  │  节省: 平均 65%                                     │
  ├────────────────────────────────────────────────────┤
  │  Layer 2: 中间层（命令/工具输出）                     │
  │  优化工具: RTK                                      │
  │  机制: PreToolUse Hook → 确定性过滤 → 精简结果返回    │
  │  节省: 平均 80%                                     │
  ├────────────────────────────────────────────────────┤
  │  Layer 1: 输入层（记忆文件/项目上下文）               │
  │  优化工具: caveman-compress                         │
  │  机制: Python 脚本 + Claude API → 一次压缩多次受益    │
  │  节省: 平均 46%                                     │
  └────────────────────────────────────────────────────┘
```

### 7.2 联合效益估算

以一次 30 分钟 Claude Code 会话为例：

| 层面 | 无优化 | 仅 Caveman | 仅 RTK | **两者联合** |
|------|--------|-----------|--------|------------|
| LLM 输出 | ~12,000 tokens | ~4,200（-65%） | ~12,000 | ~4,200（-65%） |
| 命令输出 | ~118,000 tokens | ~118,000 | ~23,900（-80%） | ~23,900（-80%） |
| 记忆文件加载 | ~900 tokens | ~486（-46%） | ~900 | ~486（-46%） |
| **合计** | **~130,900** | **~122,686** | **~36,800** | **~28,586** |
| **总节省** | — | ~6% | ~72% | **~78%** |

从数据可以看出：
- **RTK 的绝对节省量远大于 Caveman**——因为命令输出在总 token 中占比最大（~90%）
- **Caveman 在纯对话场景中价值更高**——当 AI 会话不涉及命令执行时，RTK 无法发挥作用
- **两者联合**的边际增量主要来自 Caveman 对 LLM 输出和记忆文件的压缩

### 7.3 联合配置指南

**安装顺序**（一次性）：
```bash
# 1. 安装 RTK
brew install rtk
rtk init --global

# 2. 安装 Caveman（Claude Code 为例）
# 按照 Caveman GitHub 仓库的安装说明添加 Skill

# 3. 压缩记忆文件
/caveman:compress CLAUDE.md
```

**日常使用**：两者自动工作，无需任何手动操作。

**调试时**：
```
# 暂停 Caveman（获取详细 LLM 回复）
> stop caveman
> /caveman off

# 检查 RTK 被过滤的完整输出（tee 模式）
# tee 日志中保存了命令失败时的原始输出

# 恢复
> /caveman full
```

### 7.4 联合使用注意事项

- 两者各自的风险仍然存在：Caveman 的解释不足 + RTK 的过滤误伤
- 调试时建议**先关闭 Caveman**（恢复详细回复），**再检查 RTK tee 日志**（获取未过滤输出），以获得完整信息
- RTK 在纯对话（无命令执行）场景下不产生任何价值，此时仅 Caveman 起作用
- 对于需要**深度推理**的复杂问题，建议临时关闭 Caveman（RTK 可保持开启，因为它不影响推理质量）

## 八、与 Brevity Constraints 论文的理论关联

arXiv:2604.00025（[Brevity Constraints Reverse Performance Hierarchies in Language Models](../paper/2026-04-29-Brevity-Constraints-Reverse-Performance-Hierarchies.md)）为 Caveman 的设计理念提供了学术背书，但同时划定了边界。

### 8.1 学术验证

论文证明：在 inverse scaling 任务上，对大模型施加简洁约束不仅**省 token**，还能**提升准确率**（+26.3pp）。这意味着 Caveman 在特定场景下是"免费午餐"——成本降低的同时性能提升。

### 8.2 适用边界

但论文也揭示了关键局限：
- Inverse scaling 问题仅占任务总量的 **~7.7%**
- 对需要深度推理的任务（竞赛数学、复杂代码、多步逻辑），简洁约束可能**严重损害性能**
- 论文的 Problem-Aware Routing 仅停留在概念层面，缺乏自动化判断机制

### 8.3 Problem-Aware Routing 的启示

结合论文启示和实际工程经验，理想的 token 优化策略不是"默认全开"，而是**根据问题复杂度动态调整**：

```
查询进入
  │
  ▼
Difficulty Router（判断问题复杂度）
  │
  ├─ 简单/直达型问题（inverse scaling prone）
  │   → Caveman Full/Ultra + RTK
  │   → 双重节省，准确率可能提升
  │
  ├─ 中等复杂度
  │   → Caveman Lite + RTK
  │   → 温和压缩，保留推理过程
  │
  └─ 复杂推理型问题
      → 关闭 Caveman + RTK 保持开启
      → 给模型充分推理空间
```

ICLR 2026 的 RADAR 论文给出了基于 Item Response Theory 的 Difficulty Router 方案，未来有望与 Caveman + RTK 结合，构成从"发现现象"到"工程落地"的完整 token 优化栈。

## 九、总结

| 维度 | Caveman | RTK |
|------|---------|-----|
| **一句话定位** | Prompt Engineering 的工程化 | 数据管道的确定性过滤 |
| **本质** | 让 LLM 自己说短话 | 在 LLM 看到之前把噪音过滤掉 |
| **最大价值场景** | 问答交互为主的编程助手 | 工具调用为主的 AI Agent |
| **需要 LLM** | 输出压缩不需要；文件压缩需要 | 完全不需要 |
| **确定性** | 不确定（依赖 LLM 遵循能力） | 确定（算法过滤） |
| **跨会话** | Hook + 状态文件 + 压缩文件 | 全局 Hook + config.toml |

**核心结论**：

1. **Caveman 和 RTK 不是竞争关系，而是互补关系**——它们作用于不同数据流（输出/输入 vs 命令输出），同时启用可实现三层 token 全面优化
2. **RTK 的绝对节省量更大**（~80% 命令输出压缩），因为命令输出在总 token 中占比最大
3. **Caveman 在纯对话场景中不可替代**——RTK 在不涉及命令执行时无法发挥作用
4. **两者都不是万能药**——Caveman 在需要深度推理时应关闭，RTK 在非文本输出时无效
5. **未来方向**是 Problem-Aware Routing：根据问题复杂度动态调整压缩策略，而非一刀切
