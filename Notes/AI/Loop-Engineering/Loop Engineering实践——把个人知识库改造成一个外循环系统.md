# Loop Engineering实践——把个人知识库改造成一个外循环系统

> 上一篇 [[Loop Engineering概念澄清——内循环、外循环与Harness Engineering的边界|把概念辨清了]]：外循环的最小内核是「自动发现 + workflow + 重复(cadence) + 跨轮记忆(state)」。这一篇是把它真的装到自己的知识库上——记录设计取舍、踩到的边界、首跑结果，以及一个比代码更重要的判断：**哪些事该让循环自动做，哪些事永远不能。**

## 一、起点：一个已经在转、但靠手推的循环

我的知识库（一个 Obsidian vault，对外是 `mindforge`）本来就有一套工作流：

- 每天用 `/daily` 写工作日记，把昨天没做完的任务滚到今天；
- 有主题了就写文章（放进 `Notes/`、`paper/`、`book/`）；
- 觉得某篇够沉淀，手动跑 `/extract-knowledge`，把概念提炼进 `wiki/`，长成知识图谱（[公开站点](https://huqianghui.github.io/mindforge/wiki/)）；
- 偶尔想知道"最近该学点什么""有什么新热点"。

问题不在缺环节，而在**每一环都得我亲手推一把**。文章写完忘了同步 README、忘了重建搜索索引；wiki 里的论断放久了没人复核；"定期复盘"基本等于"想起来才做"。这正是 Loop Engineering 要解决的形态——流程是对的，缺的是让它**自己转起来**的那层基础设施。

于是我把它正式化成一个外循环系统。这篇就是这套系统的设计说明 + 实践复盘。

## 二、核心心法：管道自动化，判断留给人

动手之前先定一条总纲，因为它决定了后面每一个自治边界怎么画。

一个外循环敢不敢"放手让它自己跑"，**取决于它的 verifier（验证信号）有多强**。我把 vault 里要做的事按验证信号分成两类：

| verifier 类型 | 例子 | 有没有 ground truth | 自治级别 |
|--------------|------|-------------------|---------|
| **确定性管道** | README 失同步、qmd 索引过期、坏图片路径、死 wikilink、Claim 年龄 > 60 天 | 有（可程序判定对错） | **L2 自动执行** |
| **语义判断** | 该提取哪些概念、这条 Claim 对不对、冲突怎么解、该学什么、哪个算热点 | 无（靠 LLM 或我主观） | **L1 只提议** |

> **铁律：循环替我跑腿，但"什么值得进入 wiki"永远是我的判断——wiki 是我的认知模型，自动污染它，等于污染我自己。**

这条线一画，整个系统的安全性就有了底：所有"可程序判定、可回滚"的杂活交给机器；所有"需要品味和判断"的事，机器只准列清单，等我拍板。

## 三、三件套：循环 ≠ 记忆 ≠ trace

很多人一听"自动循环"就想到 cron，但 cron 只是其中一件。一个能无人值守的外循环，要三样东西，而且最好是**三个互相独立的东西**：

```
   ⏰ 触发器（什么时候醒）       🧠 记忆（醒来知道上回干到哪）     📜 trace（干完留底）
   ──────────────────         ─────────────────────         ─────────────────
   系统 cron                   LOOP_STATE.md                  loop-run-log.md
        ↓                      （待办快照，会被覆盖更新）        （事件流，只追加）
   run-loop.sh                      ↑   ↓                          ↑
        ↓                      命令跑完读它、写它              每轮记 START/END/耗时/退出码
   claude -p "/loop-harvest" ──────┘
   （这一发 = "一圈"）
```

- **触发器**：cron 只是闹钟，真正的"循环体"是每次那一发 `claude -p "/loop-harvest"`——它起一个**全新的、无记忆的** agent 会话，跑完即销毁。
- **记忆**：正因为每发都是失忆的，"上次收割到几号""还有哪些候选等审"必须落到文件里。这就是 `LOOP_STATE.md` 的意义——**模型会忘，仓库不会忘**。它是跨轮记忆的载体，也是我人工审阅的唯一入口：只看这一个文件，就知道循环替我发现了什么。
- **trace**：`loop-run-log.md` 只追加不覆盖，记每一轮"哪天跑的、跑了多久、退出码几"。出问题时从这里回溯。它和 STATE 的区别是：**STATE 是"现在的待办快照"，run-log 是"历史的事件流"**。

把这三件拆开，是这套设计里我最满意的决定。混在一起的"自动化脚本"很难审计；拆开之后，触发归触发、记忆归记忆、留痕归留痕，每一层都能单独看、单独换。

## 四、循环目录：5 个 loop × 3 个 phase

参照 Cobus Greyling 的 7 个外循环 playbook，我把 vault 要跑的事收敛成 5 个循环，分 3 个 phase 上线：

| Loop | 对应 playbook | Phase | 命令 | cadence | 它干的事 | 自治级别 |
|------|--------------|-------|------|---------|---------|---------|
| **① 文章收割** | Post-Merge Cleanup | P1 | `/loop-harvest` | 每天 | 扫新文章 → 同步 README、重建 qmd 索引、修坏图；识别候选概念列清单 | 管道 L2 自动，入 wiki L1 提议 |
| **② 日终 triage** | Daily Triage | （已有） | `/daily` | 每天 | 写日记、滚动未完成任务 | L1（我手动） |
| **③ 周度复盘** | Changelog Drafter | P2 | `/loop-weekly` | 每周日 | 本周回顾：日记摘要、任务完成率、热点概念 | L1 提议 |
| **⑤ 热点雷达** | Issue Triage | P2 | `/loop-weekly`（合并） | 每周日 | 扫近 30 天领域新名词/新框架/新论文 | L1 提议 |
| **④ 图谱保鲜** | Dependency Sweeper | P3 | `/loop-hygiene` | 每周六 | 修死链坏图；挑出 stale/低置信/冲突列清单 | 死链坏图 L2，stale/冲突 L1 |

Loop③ 和 ⑤ 合并进同一个 `/loop-weekly`，因为它俩同 cadence、同为 L1 报告、共享同一块 STATE 区——没必要拆成两个命令两次唤醒。

### 一条容易踩的坑：热点雷达的约束

热点雷达最容易退化成"扫盲机"。我给它定了死规矩：

> **wiki 缺失 ≠ 我不知道。Scout 只报"近 30 天的新名词/新框架/新论文"，与已有概念交叉给关联，绝不铺基础概念。**

否则它每周都会兴高采烈地告诉我"你 wiki 里还没有 RAG 的页面哦"——而 RAG 我熟得很，只是没空建页。验证信号弱的循环，约束必须写得更死。

## 五、硬闸门：永不自动

L1/L2 是"默认行为"，但有些事我要的是**代码级的绝不**，不是"尽量别"：

- ❌ 不把任何 Claims 自动合并进 `wiki/`（入库永远靠手动 `/extract-knowledge`）；
- ❌ 不自动改 Claim 置信度结论、不裁决冲突、不删概念页；
- ❌ 不读、不处理 `personal-journal/`（私人日志，另有 L1 hook 兜底）。

这里用到一个我在别处反复验证的层级原则——**约束力要和它的强制力匹配**：

| 级别 | 存储位置 | 强制力 | 适合放 |
|------|---------|--------|--------|
| L1 确定性 | Hook（代码执行） | 100% | 安全红线（personal-journal 禁访） |
| L2 强提示 | CLAUDE.md / Rules | ~90% | 行为规则（"遇到 X 必须 Y"） |
| L3 弱提示 | Memory | ~60% | 背景知识（用户画像、项目上下文） |

"永不自动"这类红线，绝不能只写在 Memory 里指望模型自觉——它得提权到 charter 的硬闸门，必要时落成 hook。

## 六、实现：六个文件 + 三个命令

全部塞进 `.claude/loops/`（这个目录 gitignore，不污染公开仓库，也让私人 STATE/log 留在本地）：

| 文件 | 角色 |
|------|------|
| `LOOP.md` | charter：原理、自治分级、硬闸门 |
| `LOOP_STATE.md` | **状态脊柱**——所有"等我拍板"的东西都在这 |
| `loop-run-log.md` | 运行流水账（trace） |
| `loop-budget.md` | token 预算 + kill switch + early-exit |
| `run-loop.sh` | cron 调用的唯一入口 |
| `install-cron.sh` | 装 / 卸 / 查 cron 条目 |

加上 `.claude/commands/` 里的三个 slash 命令：`loop-harvest`、`loop-weekly`、`loop-hygiene`。

`run-loop.sh` 的循环体只有一句要害：

```bash
# 先查 kill switch，再跑命令，最后记一行 trace
[ -f .loop-disabled ] && exit 0
claude -p "/$CMD" --permission-mode acceptEdits >> "last-$CMD.out" 2>&1
```

`--permission-mode acceptEdits` 让文件编辑自动放行（README 加行、写 STATE），但破坏性操作仍会被拦——命令内部还有"永不自动"的硬闸门双保险。

成本护栏写在 `loop-budget.md`：单一可变面（每个 loop 只动它该动的目录）、diff-based（自动改动必须小且可审计）、失败不重试（单次失败写 trace 等下个 cadence，不在循环里 sleep-retry）、无 ground truth 不放手。还有一个一键全停：

```bash
touch .claude/loops/.loop-disabled   # 立刻停掉所有自动运行
rm    .claude/loops/.loop-disabled   # 恢复
```

## 七、首跑实录：故意先不开 cron

我没有写完就挂 cron，而是**先手动空跑**——这是分阶段信任的关键。第一次 `/loop-harvest` 的真实结果：

- 扫到 2 篇新文章；
- 确定性管道**零改动**：README 早已收录、qmd 索引已最新、5 个 wikilink 全部解析、0 坏图——文章入库时手工做得很干净，管道没活干；
- 语义层识别出 6 条候选概念，**全部写进 STATE 待审区，0 条入 wiki**；其中只有 `loop-engineering` 是够格的新原子概念，其余要么没到"被 2+ 篇引用"的门槛，要么和现有页重叠。

这次空跑验证了两件事：一是确定性管道安全（它不乱改）；二是语义层守得住边界（它不越权入库）。确认这两点，才敢 `install-cron.sh` 真正挂上定时。

## 八、一个比代码更重要的洞察：瓶颈是基础设施，不是智能

做完这套我更确信一件事：**在当前模型质量下，控制器（模型本身）已经够用了，缺的是把它接进世界的那层"身体"。**

借控制论的话说，模型是个无状态的控制器 `f(context) → output`；而这套外循环系统补的全是它**身体**的部分——

- **传感器**：verifier（死链检测、Claim 年龄）、自动发现（`find -newermt`）；
- **执行器**：connectors（qmd、git、README 写入）；
- **记忆**：`LOOP_STATE.md`；
- **触发**：cron 心跳。

为什么更聪明的模型也替代不了这层？因为这些是**系统属性，不是智能属性**：持久记忆、自我触发、对外副作用、跨轮一致——再聪明的无状态函数，也无法凭空长出一块硬盘或一个闹钟。认知脚手架（选 context、规划、自我纠错）会被更强的模型吸收，但"具身事实"不会。这也是为什么 Loop Engineering 是一门**工程**，而不是"等模型变强就自动解决"的问题。

## 九、上线节奏与原生替代

**节奏**：P1 确定性管道先上（风险最低）→ P2 纯报告（不碰 wiki，零风险）→ P3 半自动保鲜。每一阶段手动跑顺、确认安全，再交给 cron。

**触发器的两个原生替代**（如果不想手写 cron）：

- `/schedule` —— Claude Code 原生的 cron 调度远程 agent，不依赖本机开机，最接近真·无人值守；
- `/loop` —— 当前会话里按间隔重复跑，但关了终端就停，适合盯着看的短任务，不适合后台收割。

无论用哪种触发器，**记忆和 trace 仍然落在 `LOOP_STATE.md` / `loop-run-log.md`**，这一层不变。

## 十、小结：什么样的工作流值得做成外循环

不是所有事都该自动化。回看这套系统，真正适合做成外循环的工作流有三个特征：

1. **重复且有节奏**——每天/每周固定要做，否则自动化的收益盖不过维护成本；
2. **大部分步骤有 ground truth**——能交给确定性管道的越多，越敢放手；
3. **判断点能干净地隔离出来**——把"需要品味"的环节收敛成 STATE 里几条待审，人只在这几个点上花注意力。

我的 PKC 恰好三条都满足。最后留下的不是"全自动的知识库"——而是一个**帮我跑腿、把判断浓缩到一处等我拍板**的系统。这大概就是当前阶段 Loop Engineering 最务实的形态：不追求取代判断，只追求把判断之外的一切自动化掉。
