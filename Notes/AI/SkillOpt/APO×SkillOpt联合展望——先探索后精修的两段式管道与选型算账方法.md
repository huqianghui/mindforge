# APO × SkillOpt 联合展望——先探索后精修的两段式管道与选型算账方法

> 承接 [[SkillOpt实战篇：video2frames提示词调优——从agent-lightning APO移植到SkillOpt|实战篇]] §八的 100 任务配对对决：机制上更稳的 SkillOpt 并没有赢，自由探索的 APO 反而摸到了任务天花板。本篇不再把两者当对手，而是回答两个自然的后续问题：**能不能串成管道——APO 大胆探索、SkillOpt 小步精修？** 以及更重要的：**开跑之前，怎么用三个数算出这条管道值不值得跑？**
>
> 一句话定位：**这是一篇"先算账再开跑"的方法论——σ_d、门控分辨率、剩余空间三个数都能从已有 eval 产物里免费算出来，几分钟的算术可以省掉一次百万 token 的无效训练。video2frames 上把账走完的答案是"停"，且矛头指回 reward 权重与数据分布的先天错配（§五）。**

---

## 一、为什么想到联合：两者的互补性是结构性的

[[SkillOpt实战篇：video2frames提示词调优——从agent-lightning APO移植到SkillOpt|实战篇]] §一拆过两者的三层机制差异，这里换个角度看——**它们的强项和弱项恰好互补**：

| 维度 | agent-lightning APO | SkillOpt |
|---|---|---|
| 优化器动力学 | 并行随机搜索：自由改写 + beam 重排，**探索强、方差大** | 带信任域的 SGD：有界编辑 + 在位者门控，**利用强、方差小** |
| 擅长的阶段 | 初始 prompt 离最优很远时，**跳出局部邻域找盆地** | 已在好盆地内时，**单调精修不退步** |
| 失败模式 | 摆动：好规则被一次整篇重写丢掉，收敛不稳定 | 探索不足：护住的可能只是初始 skill 附近的小邻域 |
| 实测表现（video2frames） | 摸到天花板（+0.018，全部在 judge_score） | 增益未泛化（gate 过拟合 val，held-out 上与 baseline 打平） |

这正是优化领域的经典分工：**先用高方差的全局方法找盆地，再用低方差的局部方法收敛**。数值优化里叫 basin hopping / warm-start，训练里叫"大学习率粗训 + 小学习率微调"。文本空间没有理由例外。

## 二、两段式管道：机械上零成本，接口就是一个文本文件

```
      阶段 1（探索）                      阶段 2（精修）
┌─────────────────────────┐      ┌──────────────────────────────┐
│  agent-lightning APO    │      │  SkillOpt                    │
│  beam search 自由改写    │ ───> │  best_prompt.txt 作为         │
│  在 APO train/val 上筛   │      │  skills/initial.md 种子       │
└─────────────────────────┘      │  有界编辑 + 门控单调精修        │
                                 └──────────────────────────────┘
                                              │
                                              v
                                 ┌──────────────────────────────┐
                                 │  最终 held-out 配对对决        │
                                 │ （对两段都无污染）              │
                                 └──────────────────────────────┘
```

衔接成本几乎为零，因为 SkillOpt 的"模型初始化"就是一个 Markdown 文件：

- **种子替换**：把 APO 产出的 `best_prompt.txt` 拷成 SkillOpt 的 `skills/initial.md`，一行 `cp` 完事；
- **信号一致**：前提是评分逐字节兼容（实战篇的移植纪律在这里兑现价值——两段的 soft/hard 是同一把尺子）；
- **judge 不换**：judge 定义分数量纲，两段和最终对决必须用同一个 judge，否则所有跨段比较作废。

与 RL 的类比也顺理成章：APO 相当于**population-based 预训练**产出一个好的初始 policy，SkillOpt 相当于在其上做 **trust-region 微调**——每步编辑不超过 `learning_rate` 条、val 上不劣于在位者才 accept。

## 三、数据卫生：同分布 ✓，但重点是五份互斥切分

容易把要求说成"独立同分布"，只对一半。**分布要相同**（都从同一任务池分层随机采样，不能给某段偏难/偏易的样本），但更关键的是**成员互斥**——因为被优化过程"看过"的任务都会被过拟合：

- APO 阶段：prompt 是对着 APO 的 train/val 反复筛出来的，APO best 在这些任务上分数**虚高**；
- SkillOpt 阶段以 APO best 为种子坐庄门控。如果 SkillOpt 的 val 混入 APO 见过的任务，**在位者带着虚高分数守擂**，挑战者的真实改进被系统性压低——门控决策整体有偏，方向是"错杀好编辑"。

所以是五份互斥切分，全部同分布：

```
任务池 ──分层随机──> APO train │ APO val │ SkillOpt train │ SkillOpt val │ 最终 test
                    └────── 互斥，任何任务只出现在一份里 ──────┘
```

工程上就是 faceoff 已经在用的那个参数：`prepare_data.py --exclude <apo>/data/train.jsonl <apo>/data/val.jsonl`。最终 test 对两段都无污染，大小按配对检验功效算（见 §四；video2frames 的经验值是要分辨 +0.02 需要 ≥100 条）。

一个务实的推论：**两段式管道的数据预算约是单段的 1.5–2 倍**（多出一对 train/val）。任务池小的场景（几百条）可能撑不起五份互斥切分——这本身就是一条选型依据（§五）。

## 四、先算账再开跑：三个数与一个判断式

管道值不值得跑，不靠感觉，靠三个数。**全部从已有 eval 产物 `results.jsonl`（每任务一行分数）里免费算出来**，时点是 APO 结束后、SkillOpt 开跑前。

### 数 1：σ 与 σ_d —— 先用一个 5 任务的玩具例子看懂怎么算

用 prompt A（APO best）和 prompt B（baseline）在**同样 5 个任务**上各跑一次 eval，`results.jsonl` 每任务一行 soft 分数，按 task id 对齐：

| 任务 | A 的 soft | B 的 soft | 差值 d = A − B |
|---|---|---|---|
| task-1（简单：白天、快递员正面） | 0.95 | 0.92 | +0.03 |
| task-2（难：夜间、模糊） | 0.45 | 0.50 | −0.05 |
| task-3（中等） | 0.80 | 0.72 | +0.08 |
| task-4（难：树影晃动） | 0.55 | 0.58 | −0.03 |
| task-5（简单） | 0.90 | 0.83 | +0.07 |

- **σ**：对 A 列 `[0.95, 0.45, 0.80, 0.55, 0.90]` 求标准差。它度量的是**任务难度的散布**——0.95 和 0.45 差 0.5 不是 prompt 时好时坏，而是夜间模糊视频谁来都拿低分。这个散布主要由数据集决定，与 prompt 好坏关系不大（video2frames 实测 σ ≈ 0.20）。
- **σ_d**：对差值列 `[+0.03, −0.05, +0.08, −0.03, +0.07]` 求标准差。难题 task-2 上 A、B **都低**（0.45 vs 0.50），一相减"这题难"这个共同因素就抵消了——**配对比较的全部价值就是把任务难度这个最大噪声源从比较中剔除**，所以 σ_d（实测 ≈ 0.12）显著小于 σ。评分或 target 模型一换，两个数都要重估。

关键事实藏在差值列里：**单任务差值（±0.03~0.08）比真实平均优势（+0.02）大好几倍**——看单个任务分不清是真强还是碰巧，只能靠 n 个任务取平均，把不确定度按 σ_d/√n 收缩。

### σ ≈ 0.20 的构成：不是紧凑分布的抖动，是"主峰 + 低分尾"的混合

一个正当的直觉质疑：分数大多在 0.6~0.9 之间，宽度 0.3 的紧凑分布 σ 顶多 0.07~0.08，哪来的 0.20？答案是**分布根本不紧凑**。faceoff 实测 baseline 均值 0.788 但 hard = 0.59——即 **41% 的任务 soft < 0.8**，真实分布是三层混合：

- **主峰**（约 6 成）：scene/courier 全对（0.4 拿满）+ judge 0.6~0.9 → soft 落在 0.8~0.95
- **半失败**（约 3 成）：分类字段错或 judge 低分 → soft 落在 0.3~0.6
- **零分尾**（少数）：非法 JSON、内容过滤 → soft = 0（硬性零分规则）

手算验证：70% 集中在 0.88、25% 在 0.45、5% 为 0 → 均值 ≈ 0.73，方差 ≈ 0.70×0.15² + 0.25×0.28² + 0.05×0.73² ≈ 0.062 → σ ≈ 0.25，正是这个量级。**σ 的大头是"少数任务整个失败"贡献的，不是好任务之间 0.85 vs 0.88 的细微差别。**三个实用推论：

1. **σ 可以被"清理"而不只是被"平均掉"**：零分尾里与 prompt 无关的部分（内容过滤只取决于输入帧）用 `--probe-content-filter` 提前剔除，直接砍 σ，比扩 val 便宜；
2. **σ_d 受尾巴影响更小**：两个 prompt 往往在同一个半失败任务上一起失败（难度共享），配对相减后尾巴也大幅抵消；
3. **报出来的 σ 和肉眼分数范围对不上时，先画直方图**：低分尾肥说明该优先做失败模式分析（把 3 成半失败任务修好 0.1 分），而不是统计扩容（在主峰里刮 0.01）。

### 数 2：门控分辨率 δ_min —— gate 能"看见"多小的真实差异

```
SE = σ_d / √n                    n = SkillOpt 的 val 大小
δ_min ≈ 2 × SE                   （95% 置信）

反解：想分辨 δ 需要  n ≈ (1.96 × σ_d / δ)²
```

数值例（σ_d = 0.12）：

| val 大小 n | SE | δ_min（gate 可分辨下限） |
|---|---|---|
| 24（默认） | 0.024 | **0.049** |
| 100 | 0.012 | **0.024** |
| 140 | 0.010 | **0.020** |
| 300 | 0.007 | **0.014** |

小于 δ_min 的真实提升，在门控眼里与噪声无法区分——accept/reject 退化成抛硬币，训练在烧 token 原地踏步（实战篇 §五那次 149 万 token 事故的统计学版本）。

### 数 3：剩余空间 δ_remain —— 不是理论推导，是实测估计

同一次配对 eval 顺手就给出：

- **APO best − baseline 的配对差值**告诉你探索阶段已经拿走了多少（video2frames：+0.018）；
- **分量拆解**告诉你还剩多少可挖：scene/courier 精确匹配已饱和在 0.94–1.00，全部差距在 judge_score——所以二段可追加空间 ≲ 0.02。

### 判断式

```
若  δ_remain  >  2 × σ_d / √n（在你付得起的 n 下）   → 接 SkillOpt 精修，值得
若  δ_remain  ≤  δ_min                              → 停：瓶颈不在 prompt
```

### 在自己的 val 上把判断式走一遍（video2frames，val=100 实测推导）

**第一步：两次 eval 测出三个数。**在同一个 val split 上分别评 baseline 和种子候选（APO best）：

```bash
python eval.py --config configs/video2frames/default.yaml \
    --skill video2frames_env/skills/initial.md --split valid_seen \
    --cfg-options env.out_root=outputs/sigma_baseline
python eval.py --config configs/video2frames/default.yaml \
    --skill data/best_prompt.txt --split valid_seen \
    --cfg-options env.out_root=outputs/sigma_apobest
```

把两份 `results.jsonl` 按 task id 对齐：

```python
import json, statistics
def load(p): return {r["id"]: r["soft"] for r in map(json.loads, open(p))}
a = load("outputs/sigma_apobest/results.jsonl")
b = load("outputs/sigma_baseline/results.jsonl")
d = [a[k] - b[k] for k in a if k in b]
print("σ   =", statistics.stdev(a.values()))   # ≈ 0.20  任务难度散布
print("σ_d =", statistics.stdev(d))            # ≈ 0.12  优势的散布
print("δ_observed =", statistics.mean(d))      # ≈ +0.018 APO 已拿走的优势
```

可选第四个数：**同一个** skill 在 val 上评两遍、逐任务求差再算 std → σ_noise（纯 judge 噪声下限）。SkillOpt 候选都是小步编辑，候选 vs 在位者的 σ_d 介于 σ_noise 与 0.12 之间，取 0.12 是保守估计。

**第二步：门控分辨率。**val = 100 → δ_min ≈ 2 × 0.12 / √100 = **0.024**。

**第三步：剩余空间。**APO best 已 +0.018，且差距全部在 judge_score（scene 0.94~0.95、courier 0.98~1.00 已饱和）；天花板 ≈ +0.02 → **δ_remain ≈ 0.002~0.02**。

**第四步：代入判断式。**δ_remain（≤0.02）< δ_min（0.024）→ **不值得接 SkillOpt**。反过来算翻案成本：要分辨 +0.01 需要 n ≈ (1.96 × 0.12 / 0.01)² ≈ **550 条** val——每次 gate 评估 550 任务 × 约 2.8 万 prompt token/任务 × 几十步 gate，成本完全不成比例。

**这就是"先算账再开跑"的全部内容：三个数、一次配对 eval、几分钟算术。**而这个"停"字的根源，在下一节——瓶颈不是优化器，是 reward 结构。

## 五、判断式说"停"之后：reward 的先天错配、溯源与改良方案

三个数的算账把矛头指回了 reward 设计本身——`soft = 0.2·scene + 0.2·courier + 0.6·judge` 这组权重，对这个数据集的实际分布来说**从一开始就不合理**。

### 溯源：缺陷在设计当时就可发现——reward 设计需要三份输入

容易把这个错配当成"事后才能知道的事"，其实要分两层。**courier 正例 0.2%（5,847 条里 13 个）扫一遍标签就能发现**（零训练成本），发现即可推出"0.2 权重 + 对称匹配 = 常数分量"；**scene 会饱和在 0.94 确实要跑了才知道**，但一次 100 条的 baseline 校准跑只要几美元——和后来 149 万 token 的无效训练相比可以忽略。所以准确的结论是：**reward 设计不是写规格，是做实验**，它在第一天就需要三份输入：

| 输入 | 何时可得 | 决定什么 |
|---|---|---|
| **业务属性**（字段重要性、漏报/误报不对称代价、下游消费方式） | 设计时，问客户 | **验收分**的口径与 hard 阈值 |
| **数据分布**（类别均衡、正例稀疏度、难度散布） | 设计时，扫标签 | 哪些分量该约束化/重采样，权重能否兑现 |
| **基线能力**（各分量饱和度、judge 噪声 σ_noise） | 设计时，一次校准跑 | **优化 reward** 的权重分配与可分辨性 |

原始 APO reward 只用了第一份输入——权重按字段业务重要性拍定，从未对过数据分布和基线饱和度。移植时的逐字节兼容（为可比性，本身正确）忠实继承了这个缺陷，直到 faceoff 实测才照出来。实战篇的"客户确认六问"已隐约意识到问题，但把它们全当成**问客户的问题**——更深一层的教训是：其中大半应该是**开跑前的测量动作**，答案在数据里，不在客户嘴里。

### 错配的实测证据

| 分量 | 权重 | 实测状态 | 对优化的实际贡献 |
|---|---|---|---|
| scene_match | 0.2 | 0.94~0.95，饱和 | 接近常数，无梯度信号 |
| courier_match | 0.2 | 0.98~1.00，饱和 | 正例仅 0.2%（5,847 条里 13 个，各切分正例数为 0），实际度量的是"避免误报"——一个常数天花板 |
| judge_score | 0.6 | 0.66~0.70，唯一有差异的分量 | 承载全部可优化空间，却对三个文本字段只出**一个合并分** |

后果是信噪比的双向恶化：

- **信号被压缩**：0.4 的权重是"死权重"——两个分类分量在所有参赛者之间几乎恒定，任何 prompt 改进只能通过 judge 那 0.6 折算进 soft，可观测的 δ 被结构性打了六折；
- **噪声占比升高**：σ_d 里剩下的几乎全是 judge 噪声（合并打分粒度粗、语义评分天然抖动），死权重不贡献差异，但也不稀释噪声。

分子（δ）变小、分母（σ_d）不降——**δ_remain ≤ δ_min 这个"停"字，一半是任务天花板，另一半是 reward 结构自己造成的**。

### 改良方案：两本账分家，一个让 APO 和 SkillOpt 都重获空间的 reward v2

用一个"按业务直觉加权"的函数同时充当验收分和训练信号，是原始设计的根本缺陷。改良的第一原则是**分家**：

- **验收分**：原公式逐字节不动——历史 run 可比性、业务汇报口径都不破坏，最终 faceoff 仍用它裁决；
- **优化 reward（v2）**：只喂给训练循环（APO 的候选排序 / SkillOpt 的门控与反思桶），完全按信噪比设计：

```
soft_opt = 0.45 × judge_detail      # english_detail：事件覆盖 / 无幻觉 / 时序正确（下游主消费）
         + 0.20 × judge_brief       # 主体+动作提炼准确、≤20 词
         + 0.10 × judge_title       # 抓最关键事件、≤6 词
         + 0.25 × rule_compliance   # 确定性规则分（见④，零噪声）

乘法约束（不占权重，只设上限）：
  scene 错       → soft_opt ×= 0.5
  courier 误报   → soft_opt ×= 0.3    # 若正例进入切分：漏报 ×= 0.2（不对称，比例找客户确认）
  非法 JSON      → 0        内容过滤 → 剔除不计分
```

四个设计决策，逐一对应实测病灶：

1. **拆分 judge（0.45/0.20/0.10）**：一个合并分变三个半独立分——反思阶段能看到"哪个字段错了"（error 桶按字段分类），且多分量平均后 judge 噪声按约 1/√3 收缩，**分母 σ_d 直接降**；
2. **饱和分量约束化**：scene/courier 从加权项改为乘法上限——语义从"继续加分"变成"不许退步"，腾出的 0.4 死权重全部还给有信号的分量，**分子 δ 恢复全额通过**；
3. **courier 诚实处理**：业务关键 → 把 13 个正例过采样进 train/val + 不对称惩罚（漏报比误报罚得重）；不关键 → 只留误报约束，不再假装它有 0.2 的话语权；
4. **新增 rule_compliance（0.25，白送的零噪声信号）**：initial.md 里本来就有硬性风格规则——词数上限、用 "person" 不用性别词、禁止提 camera/frames、Non-Notable 触发句式——全部可用正则/词数**确定性打分**。现状是 prompt 要求它们但 reward 不测量它们；纳入后这 0.25 是完全无噪声的梯度素材，对两个优化器都是免费的可优化空间。

再加一个便宜的降噪杠杆：用 σ_noise（同一 skill 评两遍的差值 std）判断 judge 噪声占 σ_d 的比例，占大头就让 judge 每字段打 3 次取中位数——3 倍 judge 调用换 √3 噪声收缩，通常比扩 val 划算得多。

### 落到本场景：可直接实现的完整规格

**rule_compliance 检查清单**（正则/词数即可算，每项 0/1 取均值——依据全部是 initial.md 里已有但 reward 从未测量的规则）：

| 检查项 | 依据 |
|---|---|
| JSON 合法且恰好五键 | 输出契约 |
| detail ≤ 50 词 / brief ≤ 20 词 / title ≤ 6 词 | 各字段长度上限 |
| 无性别词（man/woman/…，须用 person） | 风格规则 |
| 不提 camera / frame / timestamp | 风格规则 |
| 无"描述缺席"句式（no people / nothing visible…） | 风格规则 |
| Non-Notable 判定正确：GT 以触发句开头时输出精确前缀命中，反之不误触发 | Step 4 触发规则，GT 可判 |

**judge 分字段 rubric**（temperature=0，各出 reason + 0~1 分，每字段 3 次取中位数）：

- `judge_detail`：对照 GT 的事件覆盖率、无幻觉主体/动作、时序正确；
- `judge_brief`：主体 + 动作与 detail 一致、语气与事件严重度匹配；
- `judge_title`：是否抓住最关键事件。

**courier 的场景化处理**：本数据集所有切分正例数为 0——13 个正例连统计评估都撑不起，过采样进 val 反而会让个位数样本扭曲 gate。更诚实的做法：把 13 条正例做成**独立 canary 回归套件**，每个候选 skill 全量跑一遍、单独汇报召回，**不进 soft_opt**；主 reward 里 courier 只留误报约束（×0.3）。

**hard_opt 重校准**：量纲变了，0.8 阈值作废——新 baseline 跑完看 soft_opt 分布，按业务盲选出的"可交付线"对应分位数重定；hard 仍只用于反思分桶与业务汇报，不做 gate。

### 对两个优化器分别意味着什么

| | 改良前 | 改良后（预期） |
|---|---|---|
| 可观测 δ（分子） | 全部改进经 judge 0.6 折算 | 全额通过 + rule 分量新增空间 |
| σ_d（分母） | ≈ 0.12，几乎全是合并 judge 噪声 | 拆分 + 中位数 → 预期 ≈ 0.06 |
| **APO** | 兄弟候选重排在噪声里翻转 → 摆动 | 排序信噪比翻倍、textual gradient 有字段级素材，beam 收敛更稳 |
| **SkillOpt** | val=100 时 δ_min ≈ 0.024 > δ_remain → gate 抛硬币 | δ_min ≈ 0.012，门控重新可分辨，reject 重新有意义 |

### 改完必须重走三个数

新 reward 下 σ、σ_d、δ_remain 全部失效：重录 baseline、按 §四流程重测。预期是**分子放大 + 分母减半**——判断式从"停"翻转为"值得跑"，这时再上 §二的两段式管道乃至 §七的更紧集成，才有统计学意义。

### 设计好之后：怎么和业务沟通——价值判断归业务，数学归工程

把 `0.45 × judge_detail` 拿给业务看是无效沟通——权重是工程推导的结果，**价值排序才是业务的输入**。业务真正要拍板的只有四件事：

| 要业务决定什么 | 怎么问（他们的语言） | 落到 reward 哪里 |
|---|---|---|
| 不对称代价 | "漏掉一个真快递员，和误报一次空场景，哪个更糟？糟几倍？" | courier 约束惩罚比例 |
| 字段消费 | "推送后用户实际读的是标题、摘要还是详情？投诉集中在哪？" | judge 拆分权重 0.45/0.20/0.10 |
| 可交付线 | "这条输出你敢直接推给终端用户吗？"（给样例，答是/否） | hard 阈值 |
| 质量锚点 | 10~20 对输出**盲选 A/B**："哪个更好？" | 校准 judge + 反推权重 |

第四条是最有效的一招：**不问权重、问偏好，从偏好反推权重**。业务说不出"detail 该占 0.45"，但对着两条真实输出选 A/B 毫不含糊；judge 排序与业务盲选的一致率 ≥ 85%，judge 才获得"代理业务"的资格——此后训练循环里的百万次打分都不用业务再看。

沟通载体与时点，两条纪律：

1. **样例锚定量纲，不是公式**：给出"0.9 / 0.7 / 0.4 各长什么样"的分数阶梯样例，业务对分数有了体感，"平均分 0.78 → 0.81"才有意义。汇报永远用验收分和 hard 比率（"可直接交付比例从 59% 到 X%"），soft_opt 是工程内部量纲不出场——两本账在沟通层同样分家。
2. **一次谈透、训练前锁版**：大规模训练后再改分数 = 重付一遍训练成本。四件事 + 盲选校准在开跑前一次谈完、落字为凭（如 `doc/reward-design.zh.md` 加一节"业务确认记录"）；此后 reward 任何改版都重触发同一流程——这就是"分数对话是全项目最便宜的保险"的操作化版本。

### 设计稿 ≠ 定稿：三个确认动作与一小时共商流程

上面的 reward v2 是**设计稿**——结构是定的（judge 拆分 + 乘法约束 + 确定性规则分 + canary 套件，每条对应一个实测病灶），但权重数字（0.45/0.20/0.10/0.25）和惩罚比例（×0.3/×0.2）是工程按"下游主消费 detail + rule 零噪声"推理给的**初始值**。设计稿到定稿之间隔着三个确认动作，缺一不可：

| 待办 | 谁定 | 定什么 | 通过标准 |
|---|---|---|---|
| 盲选 A/B 校准（10~20 对真实输出） | 业务 | 验证 judge 排序符合真实偏好，反推 0.45/0.20/0.10 是否需微调 | judge 与业务盲选一致率 ≥ 85% |
| 不对称代价确认 | 业务 | courier 漏报 ×0.2 / 误报 ×0.3 目前是拍的——要问出真实倍数 | 比例落字进确认记录 |
| 新 baseline 校准跑 | 工程 | hard_opt 分位数阈值 + 重测 σ、σ_d、δ_remain 三个数 | 判断式翻转为"值得跑" |

前两项要业务到场，与其零散拉会，不如一次一小时的**共商工作坊**，脚本如下：

1. **开场（10 分钟）——讲两本账，不讲公式**："你们看的验收分和汇报口径一个字不改；我们内部换一把训练尺子，让优化器看得更清。"业务的抵触多半来自"分数要变"的误解，先拆掉。
2. **分数阶梯（10 分钟）——建立体感**：给出 0.9 / 0.7 / 0.4 各一条真实输出样例，让业务对"分数长什么样"有锚点。
3. **盲选 A/B（20 分钟）——最重的一环**：现场过 10~20 对输出，只问"哪个更好"，不解释来源；逐对记录。这 20 分钟买到的是此后训练循环里百万次打分的代理资格。
4. **四个拍板问题（15 分钟）**：按前表逐个问——漏报误报糟几倍、用户实际读哪个字段、这条敢不敢直接推给用户（配样例）、投诉集中在哪。
5. **收尾（5 分钟）——落字与锁版**：确认记录当场写进 `doc/reward-design.zh.md`"业务确认记录"一节，并宣布规则：此后 reward 任何改版都重新触发本流程。

会后工程侧闭环：算一致率（<85% 先修 judge rubric 再来）→ 按盲选结果微调权重 → 跑新 baseline 校准 → 重走 §四三个数 → 判断式通过才锁版开训。**业务一小时的投入，换掉的是训练完才发现优化错方向的整轮返工。**

一句话收束：**验收分按业务定，优化 reward 按分布和基线定，两本账从第一天就分开——reward 权重必须对着数据分布定，而不是对着字段的业务重要性拍脑袋；而业务的角色是提供价值排序和质量锚点，不是审公式。**

## 六、选型依据：一张决策表

把机制分析（实战篇 §一）、实测对决（§八）和上面的算账方法收拢成可执行的选型规则：

| 场景特征 | 推荐 | 理由 |
|---|---|---|
| 初始 prompt 粗糙、离最优明显很远（baseline 分数低、失败模式五花八门） | **只跑 APO** 或管道阶段 1 | 探索红利最大，摆动的代价相对可接受 |
| 已有一个还不错的 prompt，需要在生产上**可控地增量演进**（不退步 > 涨得快） | **只跑 SkillOpt** | 门控的单调性保证正是为此设计 |
| APO 摸到一个好盆地，且算账显示 δ_remain > δ_min | **两段式管道** | 探索找盆地 + 精修收敛，各用所长 |
| 算账显示 δ_remain ≤ δ_min（付得起的 val 下） | **都不跑** | 瓶颈在 target 模型或 reward 结构（§五），先改再回来 |
| 任务池太小（撑不起五份互斥切分） | 单段 + 大 held-out | 数据不够时，管道的第二段只会继承污染 |
| 评估噪声大（σ_d 高）且 judge 不稳定 | 先修 eval 再谈优化 | σ 不缩小，换什么框架都躲不开（实战篇结论 4） |

两条通用纪律，不随场景变：

1. **裁判唯一且中立**：同一个 judge、对所有阶段无污染的最终 held-out、配对分析——没有这三样，任何"管道有效"的结论都不成立。
2. **每段结束都要过一遍三个数**：APO 结束算一次决定要不要接 SkillOpt；SkillOpt 结束（如果跑了）在最终 held-out 上再对决一次确认增益泛化——实战篇的教训就是 val 上的 +0.008 在 70 个新任务上蒸发。

## 七、展望：串行之外的两种更紧集成

两段式串行是最保守的形态，还有两个方向值得探索（均为设想，尚未实测）：

**① 重启式（restart）**：SkillOpt 训练出现"后期 epoch 从不提升"的信号时（实战篇 §七的症状表），不是加预算，而是触发一次 APO 式的**自由重写爆发**——从当前 skill + step buffer 里积累的失败模式生成 K 个候选重写，gate 从中选优后继续小步精修。相当于模拟退火的升温重启，SkillOpt 现有的 `lr_control_mode="none"`（full rewrite 模式）已经有这个机制的影子，缺的是"何时触发"的调度器。

**② 内嵌式（interleaved）**：把 beam search 降级为 SkillOpt 的一个 epoch 级探索层——每个 epoch 末（slow update 的同一时点）并行生成多个候选方向，用小规模 sel-set 粗筛后只把最优者交给下一 epoch 的门控主线。探索预算随训练进程 cosine 衰减，前期多探索、后期纯精修。这本质上是把"APO 的种群"塞进"SkillOpt 的谱系"，代价是 gate eval 成本乘以 beam 宽度——所以更适合 rollout 便宜的任务。

判断这些集成值不值得做的标尺，仍然是 §四那三个数：**探索机制的价值 = 它能打开多少新的 δ_remain；精修机制的价值 = 它能否在付得起的 n 下分辨这些 δ。** 机制设计可以千变万化，算账方法不变。
